# 7.1 引言：大模型推理的性能瓶颈

大模型自回归推理过程中，Transformer 架构的 Attention 机制是核心计算环节，也是性能瓶颈的主要来源。随着模型参数量（如 70B、175B）与推理上下文长度（如 8k、16k）的提升，传统推理系统的内存管理与任务调度方式已无法适配高并发、低延迟的推理需求。vLLM 0.16.0 版本通过 PagedAttention 分页 KV 缓存管理与连续批处理（Continuous Batching）两大核心技术，突破传统推理的性能限制，实现吞吐量与延迟的双重优化，成为高性能大模型推理的核心解决方案。

## 7.1.1 KV Cache 在自回归生成中的作用

在大模型自回归生成过程中，每一轮 Token 生成都需要基于历史 Token 计算 Attention 分数，而 Attention 机制的核心计算是 Query（Q）与 Key（K）的矩阵乘法，Value（V）则用于最终的特征聚合。由于历史 Token 的 K 和 V 张量在后续生成过程中会被重复使用，若每次生成新 Token 都重新计算所有历史 KV 张量，会导致计算量呈指数级增长，严重降低推理效率。

KV Cache（Key-Value Cache）正是为解决这一问题而设计的优化机制，其核心作用是缓存每一轮生成过程中产生的 K 和 V 张量，后续生成新 Token 时，仅需计算新 Token 的 Q 与历史缓存的 K 的 Attention 分数，无需重复计算历史 KV 张量。根据 vllm 0.16.0 官方文档，KV Cache 可将自回归推理的计算量降低 90% 以上，是大模型推理提速的核心基础。

需要注意的是，KV Cache 的显存占用量与上下文长度、请求数量正相关，随着推理请求的增多和上下文的延长，KV Cache 会快速占用大量 GPU 显存，成为制约推理吞吐量的关键因素。

## 7.1.2 传统推理系统的内存管理挑战

传统大模型推理系统（如 Hugging Face Transformers）采用“静态预分配”的 KV Cache 管理方式，即针对每个推理请求，提前分配固定大小的显存空间用于存储 KV Cache，这种方式在高并发、长上下文场景下会面临三大核心挑战，这也是 vllm 0.16.0 文档中明确指出的传统架构痛点：

1. **KV Cache 随请求数线性增长**：每个推理请求都需要独立分配 KV Cache 空间，显存占用量与请求数量呈线性正相关。在高并发场景下（如同时处理数十、上百个请求），KV Cache 会快速耗尽 GPU 显存，导致无法处理更多请求，甚至触发 OOM（Out of Memory）错误。

2. **预分配导致的内存碎片问题**：
        

    - 内部碎片：为适配最长上下文长度，预分配的 KV Cache 空间往往大于实际需求，未被使用的部分即为内部碎片，造成显存浪费；

    - 外部碎片：多个请求的 KV Cache 空间分散分配，导致显存中出现大量不连续的空闲空间，虽然总空闲空间足够，但无法满足新请求的连续空间分配需求，进一步降低显存利用率。

3. **高并发场景下的显存压力**：传统推理系统不支持 KV Cache 的动态共享与复用，每个请求都独占一块显存空间，即使不同请求的历史上下文存在重叠，也无法共享 KV Cache 数据，导致显存利用率极低。在高并发场景下，显存压力急剧增加，推理延迟显著上升，甚至出现请求超时、服务中断等问题。

## 7.1.3 vLLM 的性能突破：PagedAttention + 连续批处理

针对传统推理系统的内存管理与调度瓶颈，vllm 提出了“PagedAttention + 连续批处理”的协同优化方案，实现了推理性能的跨越式提升，其核心优势的是：

- PagedAttention：借鉴操作系统虚拟内存的分页机制，将 KV Cache 划分为固定大小的块（Block），实现 KV Cache 的动态分配、回收与复用，彻底解决内存碎片问题，提升显存利用率；

- 连续批处理：突破传统静态批处理的限制，实现推理任务的迭代级动态调度，允许新请求在现有任务执行过程中动态加入，完成任务及时退出，始终保持 GPU 满载运行，大幅提升推理吞吐量。

根据 vllm 0.16.0 官方性能测试数据，该方案在相同硬件环境下，相比传统推理系统，吞吐量提升最高可达 317%，延迟降低 50% 以上，同时支持更长的上下文长度与更高的并发请求量，完美适配生产环境的推理需求。

# 7.2 PagedAttention：基于分页的 KV 缓存管理

PagedAttention 其核心设计理念源于操作系统的虚拟内存分页机制，通过“逻辑块与物理块分离”的方式，实现 KV Cache 的高效管理，解决传统 KV Cache 内存碎片、显存利用率低的问题。vllm  PagedAttention 是实现高并发、长上下文推理的基础，也是 vLLM 性能优势的核心来源之一。

## 7.2.1 设计理念：借鉴操作系统的虚拟内存分页机制

传统 KV Cache 管理采用“连续内存分配”方式，每个请求的 KV Cache 必须占用一块连续的显存空间，这也是导致内存碎片的根本原因。PagedAttention 借鉴操作系统虚拟内存的分页思想，将 KV Cache 的管理分为“逻辑层面”与“物理层面”，核心设计理念包括两点：

1. **逻辑块与物理块的分离**：
        逻辑块与物理块通过“块表”建立映射关系，请求仅需关注逻辑块的顺序，无需关心物理块的实际存储位置，实现“逻辑连续、物理离散”的存储模式。

    - 逻辑块：从用户视角，每个推理请求的 KV Cache 被划分为若干个固定大小的逻辑块，逻辑块的数量由请求的上下文长度决定（上下文长度 ÷ 块大小）；

    - 物理块：从显存视角，将 GPU 显存划分为若干个与逻辑块大小一致的物理块，物理块是 KV Cache 存储的最小单元，可独立分配、回收与复用。

2. **非连续内存存储的可能性**：由于逻辑块与物理块分离，每个请求的 KV Cache 无需占用连续的物理显存空间，可由多个离散的物理块拼接而成。这种方式彻底解决了传统连续分配导致的外部碎片问题，即使显存中存在大量离散的空闲物理块，也能通过块表映射，为新请求分配足够的 KV Cache 空间，大幅提升显存利用率。

## 7.2.2 核心数据结构

PagedAttention 依赖三大核心数据结构，实现 KV Cache 的高效管理，三者协同工作，完成逻辑块与物理块的映射、分配与回收：

1. **块（Block）：固定大小的 KV Cache 单元**块是 PagedAttention 管理 KV Cache 的最小单元，默认块大小为 16 tokens，即每个物理块可存储 16 个 Token 对应的 KV 张量数据。用户可通过配置参数 `block_size` 调整块大小，适配不同模型与推理场景（如长上下文场景可适当增大块大小）。每个块的显存占用量由模型配置决定，具体计算方式见 7.2.5 节，块的状态分为“空闲”“占用”“共享”三种，分别对应不同的使用场景。

2. **块表（Block Table）：逻辑块到物理块的映射**块表是每个推理请求的核心元数据，本质是一个数组，数组的索引对应请求的逻辑块序号，数组的值对应该逻辑块映射的物理块编号。通过块表，PagedAttention 可快速查询每个逻辑块对应的物理块位置，实现 KV Cache 数据的快速读取与写入。例如，一个上下文长度为 32 tokens 的请求，若块大小为 16 tokens，则该请求的逻辑块数量为 2，块表中会存储两个物理块编号，分别对应存储前 16 个 Token 和后 16 个 Token 的 KV 数据。

3. **空闲块队列（Free Block Queue）：可用的物理块池**空闲块队列是一个线程安全的队列，用于管理所有未被占用的物理块。当新请求需要分配 KV Cache 空间时，PagedAttention 从空闲块队列中取出相应数量的物理块，分配给该请求，并更新块表；当请求完成后，其占用的物理块会被回收，重新加入空闲块队列，供后续请求复用。vllm 0.16.0 中，空闲块队列支持优先级排序，可根据物理块的存储位置（如 GPU 显存、CPU 内存）调整优先级，优先分配 GPU 显存中的空闲块，提升推理速度。

## 7.2.3 内存分配与释放流程

PagedAttention 采用“按需分配、及时回收”的内存管理策略，结合块表与空闲块队列，实现 KV Cache 内存的高效分配与释放，整个流程完全贴合 vllm 0.16.0 官方文档的实现逻辑，分为分配与释放两个核心阶段：

### 1. 按需分配（On-demand Allocation）

按需分配是指仅在需要生成新 Token 时，为请求分配对应的 KV Cache 物理块，而非提前分配全部空间，具体流程如下：

1. 用户发起推理请求，指定上下文长度与生成 Token 数量；

2. PagedAttention 根据上下文长度与块大小，计算所需的逻辑块数量（逻辑块数量 = 上下文长度 ÷ 块大小，向上取整）；

3. 从空闲块队列中取出对应数量的空闲物理块，若 GPU 显存中无足够空闲块，可将部分 CPU 内存中的空闲块分配（内存卸载）；

4. 更新该请求的块表，建立逻辑块与物理块的映射关系；

5. 生成新 Token 时，仅需为新 Token 分配对应的物理块，无需分配全部上下文的 KV Cache 空间，进一步降低显存占用。

### 2. 块回收与复用机制

当推理请求完成（达到 max_tokens 或生成 EOS Token）时，PagedAttention 会及时回收该请求占用的所有物理块，重新加入空闲块队列，供后续请求复用，具体流程如下：

1. 推理请求完成，引擎层触发资源回收指令；

2. PagedAttention 遍历该请求的块表，获取所有占用的物理块编号；

3. 将这些物理块的状态标记为“空闲”，并加入空闲块队列；

4. 清空该请求的块表，释放元数据占用的内存；

5. 后续新请求分配物理块时，优先复用这些回收的空闲块，提升显存利用率。

根据 vllm 0.16.0 官方测试，该机制可将 KV Cache 显存利用率提升至 85% 以上，相比传统预分配方式（利用率通常低于 50%），显存利用率提升显著。

## 7.2.4 写时复制（Copy-on-Write）机制

写时复制（Copy-on-Write，COW）是 PagedAttention 针对并行采样、束搜索（Beam Search）等场景设计的优化机制，核心目标是减少 KV Cache 的冗余存储，进一步降低显存占用。vllm 0.16.0 官方文档明确指出，COW 机制可在并行采样场景下，将 KV Cache 显存占用降低 50% 以上，同时不影响推理精度。

### 1. 并行采样和束搜索场景下的块共享

在并行采样（如 top-k 采样、nucleus 采样）或束搜索场景中，同一个推理请求会生成多个候选 Token 分支，每个分支都需要基于相同的历史上下文进行推理，即不同分支的历史 KV Cache 数据完全相同。传统方式下，每个分支会独立分配一套 KV Cache 空间，导致大量冗余存储。

PagedAttention 的 COW 机制允许不同分支共享相同的历史 KV Cache 物理块，具体实现为：多个分支的块表映射到相同的物理块编号，此时物理块的状态标记为“共享”，所有分支共享该物理块的 KV 数据，无需重复存储，大幅降低显存占用。

### 2. 分支写入时的块复制策略

当某个分支需要生成新 Token，进而修改 KV Cache 数据时（即写入操作），PagedAttention 会触发写时复制策略，具体流程如下：

1. 检测到分支对共享物理块的写入操作；

2. 从空闲块队列中取出一个新的空闲物理块；

3. 将原共享物理块的 KV 数据复制到新的物理块中；

4. 更新该分支的块表，将对应的逻辑块映射到新的物理块；

5. 原共享物理块的状态保持“共享”，供其他未写入的分支继续使用。

这种策略既保证了不同分支的 KV Cache 数据独立性，又避免了不必要的冗余复制，仅在需要写入时才进行复制，最大限度降低显存占用与计算开销。

## 7.2.5 块大小计算

PagedAttention 的块大小直接影响显存利用率与推理性能，vllm 0.16.0 官方文档给出了块大小的标准计算公式，同时明确了块大小对系统性能的影响，帮助用户根据实际场景优化配置。

### 1. 计算公式

每个物理块的显存占用量（单位：字节）计算公式如下：

 $block\_memory = 2 \times block\_size \times num\_kv\_heads \times head\_size \times dtype\_bytes$ 

各参数含义（参照 vllm 0.16.0 官方定义）：

-  $block\_size$ ：块大小（单位：tokens），默认 16 tokens；

-  $num\_kv\_heads$ ：KV 注意力头数量，通常与模型的注意力头数量一致，部分模型（如 LLaMA 2）支持 KV 头数量减半（num_kv_heads = num_heads / 2）；

-  $head\_size$ ：每个注意力头的维度，由模型配置决定（如 LLaMA 2 7B 的 head_size 为 128）；

-  $dtype\_bytes$ ：KV 缓存的数据类型字节数，如 FP16 为 2 字节、FP8 为 1 字节、INT8 为 1 字节；

- 系数 2：表示每个 Token 对应一组 K 和 V 张量，因此需要乘以 2。

示例：LLaMA 2 7B 模型，block_size=16，num_kv_heads=32，head_size=128，dtype=FP16（dtype_bytes=2），则单个块的显存占用为：2 × 16 × 32 × 128 × 2 = 262144 字节（256 KB）。

### 2. 块大小对内存利用率和性能的影响

块大小的选择需要在显存利用率与推理性能之间权衡，vllm 0.16.0 官方文档给出了明确的优化建议：

- **块大小过小**：
       

    - 显存利用率高：小块可更好地适配短上下文请求，减少内部碎片；

    - 性能下降：块数量增多，块表映射开销增大，同时 KV Cache 读取时的内存访问次数增加，导致推理延迟上升。

- **块大小过大**：
        

    - 性能提升：块数量减少，块表映射开销降低，内存访问效率提升，推理延迟下降；

    - 显存利用率低：大块易产生内部碎片，尤其是短上下文请求，会浪费大量显存空间。

vllm 0.16.0 推荐默认块大小为 16 tokens，该配置在大多数场景下可实现显存利用率与推理性能的平衡；对于长上下文（如 16k）推理场景，可将块大小调整为 32 或 64 tokens，提升推理性能；对于短上下文、高并发场景，可将块大小调整为 8 tokens，提升显存利用率。

# 7.4 连续批处理（Continuous Batching）

连续批处理（Continuous Batching）是 vLLM 0.16.0 实现高吞吐量推理的核心调度技术，与 PagedAttention 深度协同，突破传统静态批处理的局限性，实现推理任务的动态调度，最大化 GPU 利用率。vllm 0.16.0 官方文档指出，连续批处理是 vLLM 相比传统推理系统吞吐量提升的关键因素之一，尤其适合高并发、混合长度请求的场景。

## 7.4.1 传统静态批处理的局限性

传统大模型推理系统采用“静态批处理”方式，即先收集一定数量的推理请求，组成一个批处理（Batch），然后一次性提交给 GPU 执行，直到该批处理中所有请求都完成推理，再处理下一批请求。这种方式在实际生产场景中存在两大核心局限性：

1. **等待批处理填满的延迟**：为了提升 GPU 利用率，静态批处理需要等待批处理大小达到预设值（如 32、64）后才开始执行，若请求量不足，会导致请求长时间等待，增加端到端延迟。例如，预设批大小为 32，若当前仅收到 10 个请求，这些请求需要等待后续 22 个请求才能组成批处理，等待时间会显著增加。

2. **请求完成后的 GPU 空闲**：同一批处理中的请求，其上下文长度与生成 Token 数量不同，导致不同请求的完成时间存在差异。当批处理中某个请求完成推理后，其占用的 GPU 资源会被释放，但由于静态批处理无法动态调整批大小，剩余请求仍在执行，导致 GPU 部分资源空闲，无法充分利用。例如，一批 32 个请求中，有 10 个请求先完成，剩余 22 个请求继续执行，此时 GPU 部分计算核心与显存资源空闲，降低了整体利用率。

传统静态批处理的局限性，导致其在高并发、混合长度请求场景下，无法兼顾吞吐量与延迟，难以适配生产环境的推理需求。

## 7.4.2 连续批处理的核心理念

连续批处理的核心理念是“迭代级调度”，即每一轮 Token 生成（每一步 step）都重新选择当前可执行的请求，组成新的批处理，实现请求的动态加入与退出，始终保持 GPU 满载运行。vllm 0.16.0 中，连续批处理的核心理念包括三点：

1. **迭代级调度：每一步（step）都重新选择批处理**：与传统静态批处理“一次批处理，全程不变”不同，连续批处理在每一轮 Token 生成后，都会重新筛选当前可执行的请求（包括未完成的请求和新加入的请求），组成新的批处理，提交给 GPU 执行。这种方式无需等待批处理填满，每一步都能充分利用 GPU 资源。

2. **请求的动态加入与退出**：
        

    - 动态加入：新发起的推理请求，无需等待现有批处理完成，可在任意一步 Token 生成后，加入到下一轮的批处理中，立即执行；

    - 动态退出：当某个请求完成推理（达到 max_tokens 或生成 EOS Token）时，会在当前 step 结束后，立即退出批处理，释放占用的 GPU 资源，供新请求使用。

3. **始终保持 GPU 满载运行**：通过迭代级调度与动态加入/退出机制，连续批处理可根据 GPU 资源状态，实时调整批处理大小，确保每一步 Token 生成时，GPU 计算核心与显存资源都处于满载状态，最大化 GPU 利用率，提升推理吞吐量。

## 7.4.3 请求队列管理

为实现连续批处理的动态调度，vllm 0.16.0 采用“双队列”管理机制，即等待队列（Waiting Queue）与运行队列（Running Queue），两个队列协同工作，完成请求的调度与生命周期管理，具体实现完全遵循官方文档规范：

1. **等待队列（Waiting Queue）**：
        等待队列用于存储新发起的、尚未开始执行的推理请求，队列中的请求按优先级排序（支持用户自定义优先级）。当 GPU 有空闲资源时，调度器会从等待队列中取出适量请求，加入运行队列，参与下一轮的批处理。等待队列支持动态调整优先级，高优先级请求（如紧急推理请求）可优先被取出，加入运行队列，确保核心业务的低延迟。

2. **运行队列（Running Queue）**：
        运行队列用于存储正在执行推理的请求，这些请求会参与每一轮的批处理，生成新的 Token。每一轮 Token 生成后，调度器会检查运行队列中的每个请求，判断是否完成推理：

    - 未完成的请求：继续留在运行队列中，参与下一轮批处理；

    - 已完成的请求：从运行队列中移除，触发资源回收（KV Cache 物理块回收、计算资源释放）。

3. **已完成请求的资源释放**：
        当请求从运行队列中移除后，vllm 会立即回收该请求占用的所有资源，包括：资源回收的及时性，确保了 GPU 资源能够快速复用，进一步提升连续批处理的吞吐量。

    - KV Cache 资源：通过 PagedAttention 回收该请求占用的物理块，重新加入空闲块队列；

    - 计算资源：释放该请求占用的 GPU 计算核心与显存空间；

    - 元数据资源：清空该请求的块表、采样参数等元数据，释放内存。

## 7.4.4 与 PagedAttention 的协同

连续批处理与 PagedAttention 是 vLLM 两大核心技术，二者深度协同，缺一不可，共同实现高吞吐量、低延迟的推理。其协同机制主要体现在两个方面，完全贴合官方文档的设计逻辑：

1. **动态批处理时的 KV 块分配**：
       连续批处理的动态加入/退出机制，要求 KV Cache 能够快速分配与回收，而 PagedAttention 的分页管理的正好满足这一需求。当新请求从等待队列加入运行队列时，PagedAttention 可快速从空闲块队列中分配物理块，建立块表映射，无需等待连续的显存空间；当请求退出运行队列时，PagedAttention 可立即回收物理块，供新请求复用。这种协同机制，确保了连续批处理能够快速调整批处理大小，同时避免显存碎片问题，提升显存利用率。

2. **批处理大小与内存预算的平衡**：
        vllm  支持配置 GPU 显存预算（Memory Budget），用于限制 KV Cache 占用的最大显存空间。连续批处理的调度器会与 PagedAttention 实时通信，获取当前空闲块队列的状态（空闲物理块数量、显存占用量），动态调整批处理大小：这种平衡机制，确保了连续批处理在提升吞吐量的同时，不会超出显存预算，保障系统稳定运行。

    - 当空闲物理块充足、显存预算充足时，调度器会从等待队列中取出更多请求，增大批处理大小，提升吞吐量；

    - 当空闲物理块不足、显存接近预算上限时，调度器会减少批处理大小，暂停从等待队列中取出新请求，避免 OOM 错误。

## 7.4.5 性能对比

根据 vllm性能测试数据，连续批处理（结合 PagedAttention）相比传统静态批处理，在吞吐量、延迟、稳定性等方面均有显著优势，测试环境为：GPU 为 NVIDIA A100 40GB，模型为 LLaMA 2 7B，上下文长度为 2048，生成 Token 数量为 128，对比结果如下：

1. **与传统批处理的吞吐量对比**：
        连续批处理的吞吐量相比传统静态批处理，提升幅度为 150%~317%，具体取决于请求并发量与请求长度。其中，在高并发场景下（并发请求数 ≥ 64），吞吐量提升最高，达到 317%；在中等并发场景下（并发请求数 32~64），吞吐量提升约 220%；在低并发场景下（并发请求数 ≤ 16），吞吐量提升约 150%。核心原因是连续批处理始终保持 GPU 满载运行，避免了传统静态批处理的等待延迟与 GPU 空闲问题。

2. **混合长度请求下的稳定性优势**：
在实际生产场景中，推理请求的上下文长度往往是混合的（如部分请求上下文长度为 512，部分为 2048），传统静态批处理在这种场景下，会出现批处理大小波动、GPU 利用率不稳定的问题，导致延迟波动较大。连续批处理结合 PagedAttention，可动态适配混合长度请求，每一步都根据请求的实际长度分配 KV Cache 资源，调整批处理大小，确保 GPU 利用率稳定在 90% 以上，延迟波动控制在 ±10% 以内，相比传统静态批处理（延迟波动 ±30% 以上），稳定性提升显著。

此外，连续批处理与 PagedAttention 协同工作时，还能支持更长的上下文长度（如 16k、32k），相比传统推理系统，长上下文推理的延迟降低 40% 以上，显存占用降低 35% 以上，进一步拓展了大模型的应用场景。
