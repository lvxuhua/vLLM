## 15.1 内置 Prometheus 指标

vLLM 内置原生 Prometheus 指标暴露功能，无需额外开发，启动服务时即可开启指标暴露，核心覆盖请求延迟分布、缓存命中率、GPU 利用率三大核心维度，满足基础监控需求，所有指标均符合 Prometheus 规范，可直接抓取。

### 15.1.1 核心指标详解
1. **请求延迟分布指标**核心用于监控推理请求的延迟表现，精准定位延迟瓶颈，适配低延迟业务场景优化：

    - `vllm_request_latency_seconds_bucket`：请求总延迟分布（直方图），按延迟区间分组，单位：秒；可用于计算 P50/P95/P99 延迟，评估服务响应性能。

    - `vllm_ttft_seconds_bucket`：首字延迟（TTFT）分布（直方图），单位：秒；核心监控用户请求到第一个 token 生成的延迟，反映服务响应速度。

    - `vllm_tpot_seconds_bucket`：每个输出 token 时间（TPOT）分布（直方图），单位：秒；监控后续 token 生成的平均延迟，反映长序列生成效率。

2. **缓存命中率指标**vLLM 基于 PagedAttention 机制实现缓存优化，缓存命中率直接影响推理性能，核心指标：计算逻辑：缓存命中率 = 缓存命中次数 ÷（缓存命中次数 + 缓存未命中次数）。

    - `vllm_cache_hit_rate`：整体缓存命中率（ gauge 类型，取值 0~1）；命中率越高，推理速度越快，资源占用越低，理想值≥0.8。

    - `vllm_cache_hits_total`：缓存命中总次数（counter 类型）；

    - `vllm_cache_misses_total`：缓存未命中总次数（counter 类型）。

3. **GPU 利用率指标**核心监控 GPU 资源使用情况，避免资源浪费或过载，核心指标：

    - `vllm_gpu_utilization`：GPU 整体利用率（gauge 类型，取值 0~1）；实时反映 GPU 计算核心占用情况，理想范围 0.6~0.8（过高易导致瓶颈，过低浪费资源）。

    - `vllm_gpu_memory_usage_bytes`：GPU 内存使用量（gauge 类型），单位：字节；

    - `vllm_gpu_memory_available_bytes`：GPU 剩余内存量（gauge 类型），单位：字节。

### 15.1.3 Prometheus 配置抓取
前提：已安装 Prometheus，编辑配置文件，添加 vLLM 指标抓取任务：

```yaml
# 编辑 Prometheus 配置文件（prometheus.yml）
cat > /etc/prometheus/prometheus.yml << EOF
# 全局配置
global:
  scrape_interval:     15s     # 默认抓取间隔，可被单个 job 覆盖
  evaluation_interval: 15s     # 规则评估间隔（用于记录规则、告警等）
  external_labels:
    monitor: 'vllm-monitor'    # 可选的外部标签，可用于区分不同监控环境

# 告警配置（如果没有告警，可以留空）
alerting:
  alertmanagers:
    - static_configs:
        - targets: []
          # - alertmanager:9093   # 如果有 Alertmanager，可在此配置

# 规则文件（可包含告警规则、记录规则等）
rule_files:
  # - "first_rules.yml"
  # - "second_rules.yml"

# 抓取配置
scrape_configs:
  # 任务1：监控 Prometheus 自身
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']   # Prometheus 自身的 metrics 端口
    scrape_interval: 15s
    metrics_path: /metrics

  # 任务2：监控 vLLM 服务
  - job_name: 'vllm'
    static_configs:
      - targets: ['localhost:8000']   # 将端口替换为你 vLLM 服务的实际端口
    scrape_interval: 15s               # 抓取间隔
    metrics_path: /metrics             # vLLM 默认暴露 metrics 的路径
    # 如果需要认证或添加标签，可以在此添加 params / basic_auth / labels 等
EOF

# 重启 Prometheus 服务，使配置生效
(base) root@3e8799bf3e92:/data# prometheus --config.file=/etc/prometheus/prometheus.yml

# 验证抓取：访问 Prometheus web 界面（默认端口 9090）
# 访问 http://localhost:8000/metrics，在 Graph 页面搜索核心指标（如 vllm_gpu_utilization），确认是否抓取成功

```

## 15.2 与 Grafana 集成构建监控大盘

Prometheus 负责指标抓取与存储，Grafana 负责可视化展示，通过两者集成，构建 vLLM 全维度监控大盘，直观展示请求延迟、缓存命中率、GPU 利用率等核心指标，支持告警配置，官方推荐适配 vLLM 的监控面板模板。

### 15.2.1 集成前提

- 已安装 Prometheus（完成 15.1.3 配置，成功抓取 vLLM 指标）；

- 已安装 Grafana（推荐版本 9.0+，适配 Prometheus 数据源）。

### 15.2.2 步骤1：Grafana 添加 Prometheus 数据源

```bash
#安装grafana
rpm --import https://rpm.grafana.com/gpg.key
tee /etc/yum.repos.d/grafana.repo << EOF
[grafana]
name=Grafana
baseurl=https://rpm.grafana.com
repo_gpgcheck=1
enabled=1
gpgcheck=1
gpgkey=https://rpm.grafana.com/gpg.key
sslverify=1
sslcacert=/etc/pki/tls/certs/ca-bundle.crt
EOF


# 1. 启动 Grafana 服务（若未启动）
systemctl start grafana-server
(base) [root@foundation0 prometheus]# systemctl status grafana-server.service

# 2. 访问 Grafana  web 界面（默认端口 3000，默认账号 admin/admin）
# 3. 添加 Prometheus 数据源：
#    - 左侧菜单 → Data Sources → Add data source → 选择 Prometheus
#    - 在 URL 处输入 Prometheus 地址（如 http://localhost:9090）
#    - 点击 Save & Test，提示 "Data source is working" 即配置成功

```

## 15.3 日志配置与管理

vLLM 支持灵活的日志配置，可设置日志级别、分离访问日志与错误日志，便于问题排查、审计与运维管理，所有日志配置均通过 CLI 参数或配置文件实现，官方推荐 CLI 配置（实操便捷，适配教学）。

### 15.3.1 日志级别设置

vLLM 支持 4 种日志级别（从低到高）：DEBUG、INFO、WARNING、ERROR，默认级别为 INFO，可根据需求调整，核心参数：`--log-level <level>`。

```bash
# 1. 调试场景：设置 DEBUG 级别（输出最详细日志，含分词、推理、缓存等过程日志）
(test) [root@foundation0 model]# vllm serve Qwen2.5-0.5B-Instruct --host 0.0.0.0 --port 8000 --max-model-len 1024   --enforce-eager --uvicorn-log-level debug
之前的版本可以用--verbose

# 2. 生产场景：设置 WARNING 级别（仅输出警告和错误日志，减少日志量）
(test) [root@foundation0 model]# vllm serve Qwen2.5-0.5B-Instruct --host 0.0.0.0 --port 8000 --max-model-len 1024   --enforce-eager --uvicorn-log-level warning

# 3. 错误排查场景：设置 ERROR 级别（仅输出错误日志，聚焦故障）
(test) [root@foundation0 model]# vllm serve Qwen2.5-0.5B-Instruct --host 0.0.0.0 --port 8000 --max-model-len 1024   --enforce-eager --uvicorn-log-level error


```

### 15.3.2 访问日志与错误日志分离

默认情况下，vLLM 所有日志输出到标准输出（stdout），生产环境需分离访问日志（用户请求记录）与错误日志（服务异常记录），便于分别管理和排查，通过 shell 重定向实现：

```bash
# 启动 vLLM 服务，分离访问日志与错误日志
# 1. 创建日志存储目录（推荐）
mkdir -p /var/log/vllm

# 2. 启动服务，重定向日志（后台运行）
(test) [root@foundation0 vllm]# vllm serve Qwen2.5-0.5B-Instruct --host 0.0.0.0 --port 8000 --max-model-len 1024   --enforce-eager --uvicorn-log-level debug > /var/log/vllm/access.log 2> /var/log/vllm/error.log &

# 说明：
# 1. > /var/log/vllm/access.log：将标准输出（访问日志、正常日志）写入 access.log
# 2. 2> /var/log/vllm/error.log：将标准错误（错误日志、异常信息）写入 error.log
# 3. nohup + &：后台运行服务，避免终端关闭导致服务终止

# 日志查看命令
tail -f /var/log/vllm/access.log  # 实时查看访问日志
tail -f /var/log/vllm/error.log   # 实时查看错误日志

# 日志轮转（可选，生产推荐，避免日志过大）
# 安装 logrotate，创建配置文件
cat > /etc/logrotate.d/vllm << EOF
```

### 15.4.5 生产环境优化
- 采样率调整：生产环境建议将 `OTEL_TRACES_SAMPLER` 设为 `parentbased_always_on`，采样率设为 0.1~0.5，避免追踪数据过多占用资源；

- 数据持久化：Jaeger 数据默认存储在内存，生产环境需配置持久化存储（如 MySQL、Elasticsearch）；

- 链路筛选：通过 OpenTelemetry 配置，筛选核心请求链路，减少非关键链路的追踪数据。
