# 第十章 内存优化：从原理到实践

# 10.1 引言：为什么内存优化是大模型推理的核心挑战？

随着大模型参数规模（从百亿级到万亿级）和上下文长度（从千token到万token）的持续增长，推理阶段的内存开销成为制约模型部署效率、并发能力和响应速度的核心瓶颈。与训练阶段不同，推理阶段需兼顾低延迟、高吞吐量和高并发，且硬件资源（尤其是GPU显存）通常更为有限，因此内存优化成为大模型推理部署的关键环节——其直接决定了模型能否在目标硬件上稳定运行、能否支撑实际业务的并发需求，以及能否平衡推理性能与资源成本。

## 10.1.1 大模型推理的内存瓶颈：KV Cache 的快速增长

大模型（尤其是Transformer架构）推理过程中，内存开销主要来源于三部分：模型参数权重、KV Cache（键值缓存）和中间激活值。其中，KV Cache是导致内存瓶颈的核心因素，其内存占用随上下文长度和并发请求数量呈线性增长，成为制约大模型长上下文推理和高并发部署的关键。

具体而言，Transformer每一层的注意力机制都会为输入序列中的每个token生成对应的键（Key）和值（Value），并将其缓存起来，用于后续token的解码计算——这就是KV Cache的核心作用，通过缓存避免重复计算，降低推理延迟。但当处理长上下文（如4k、8k甚至更长token序列）或多并发请求时，KV Cache的内存占用会快速累积：假设某模型单token单头KV的内存占用为4字节，100亿参数模型（60层、128头）处理1个8k上下文的请求，仅KV Cache就需占用约8k×128×60×4×2（K和V）≈48MB；若同时处理100个并发请求，KV Cache内存占用将达到4.8GB，再叠加模型参数权重（100亿FP16参数约20GB）和中间激活值，极易超出GPU显存上限，导致推理失败或并发能力骤降。

此外，KV Cache的增长特性与请求的动态性（如不同请求的上下文长度差异、请求的启停时间不一致）结合，会进一步加剧内存管理的复杂度，引发内存碎片、缓存利用率低等问题，进一步放大内存瓶颈。

## 10.1.2 GPU显存分布：参数权重、KV Cache、中间激活的占比分析

大模型推理时，GPU显存的占用主要分为三大模块，其占比会随模型规模、上下文长度、并发量和推理策略动态变化，具体分布如下：

1. **模型参数权重**：占比相对固定，取决于模型参数规模和数据类型。例如，FP16精度下，参数占用内存=参数数量×2字节（如100亿参数≈20GB，340亿参数≈68GB）；INT8量化后，可将参数内存占用降低至FP16的1/2，INT4量化可降低至1/4。这部分内存是推理的基础，必须全程驻留GPU显存（除非使用模型卸载技术），是显存占用的“基础盘”。

2. **KV Cache**：占比动态变化，核心影响因素是上下文长度（L）、并发请求数（N）、模型层数（L_layer）和注意力头数（H），计算公式可简化为：KV Cache内存= N × L × L_layer × H × 2 × 数据类型字节数（K和V各一份）。如前文所述，当并发量或上下文长度提升时，KV Cache占比会快速上升，甚至超过参数权重，成为显存占用的“主力”。

3. **中间激活值**：占比相对较小，但波动较大，取决于输入序列长度、模型结构和批处理大小。中间激活值是推理过程中临时生成的计算结果（如Transformer层的FeedForward输出、注意力输出等），仅在单轮推理的计算过程中存在，计算完成后会被释放。但在批处理推理或长上下文推理时，中间激活值的累积也可能成为显存压力的补充因素，尤其是在未启用激活值重计算或缓存优化的场景下。

典型场景下（如100亿参数模型、FP16精度、处理4k上下文、10并发请求），GPU显存分布大致为：参数权重（20GB，约40%）、KV Cache（24GB，约48%）、中间激活值（6GB，约12%）。可见，KV Cache的占比已超过参数权重，成为内存优化的核心靶点。

## 10.1.3 内存优化的目标：提高吞吐量、降低延迟、支持更大并发

大模型推理的内存优化并非单纯“节省内存”，而是在有限的硬件资源（尤其是GPU显存）下，实现三大核心目标的平衡，最终支撑实际业务场景的需求：

1. **提高吞吐量**：在单位时间内处理更多的推理请求，降低单请求的硬件资源成本。这需要通过优化内存管理，让GPU显存能够容纳更多的并发请求，减少因内存不足导致的请求排队或拒绝，提升硬件利用率。

2. **降低延迟**：缩短单条推理请求的端到端延迟（从请求发起至结果返回），尤其是首字延迟（TTFT，Time to First Token）和迭代延迟（ITL，Iteration Time per Token），满足实时交互场景（如对话机器人、智能助手）的需求。这需要通过缓存复用、减少重计算、优化内存访问等方式，避免因内存操作导致的延迟增加。

3. **支持更大并发**：在不增加硬件成本的前提下，提升模型同时处理的请求数量；或在相同并发量下，支持更长的上下文长度，拓展大模型的应用场景（如长文档分析、多轮对话）。这需要通过高效的内存分配、碎片回收、缓存共享等技术，最大化显存的利用率。

三者之间存在一定的权衡关系（如提高吞吐量可能会轻微增加延迟，支持更长上下文可能会降低并发量），内存优化的核心就是通过合理的技术选型和参数调优，在具体业务场景中实现最优平衡。

# 10.2 PagedAttention：vLLM 的内存管理基石

vLLM 之所以能在大模型推理中实现高吞吐量和低延迟，核心在于其创新性的内存管理机制——PagedAttention（分页注意力）。该机制借鉴操作系统的虚拟内存分页思想，解决了传统注意力机制在KV Cache管理中的内存碎片、复用率低等问题，成为vLLM内存优化的核心基石。

## 10.2.1 传统注意力机制的内存管理问题

传统Transformer推理的注意力机制中，KV Cache采用“预分配+连续内存”的管理方式，即针对每个推理请求，提前分配一块与最大上下文长度（max_model_len）匹配的连续显存，用于存储该请求的所有KV Cache数据。这种方式虽然实现简单，但存在两大核心问题，严重影响内存利用率和并发能力：

### 预分配导致的内存碎片（内部碎片、外部碎片）

内存碎片分为内部碎片和外部碎片，两者均会导致显存利用率大幅下降：

1. **内部碎片**：由于预分配的内存是按最大上下文长度配置的，而实际请求的上下文长度往往小于max_model_len，导致预分配的内存中部分空间未被使用，形成内部碎片。例如，max_model_len设置为8k，但某请求仅需处理1k上下文，此时7k对应的KV Cache内存空间被浪费，且无法被其他请求复用。

2. **外部碎片**：多个请求的预分配连续内存块分散在显存中，当部分请求结束并释放内存后，会在显存中形成分散的空闲块。这些空闲块单个尺寸可能较小，无法满足新请求的连续内存分配需求（尤其是大上下文请求），导致即使总空闲显存足够，也无法分配新的请求，形成外部碎片。

在高并发场景下，内存碎片会快速累积，导致GPU显存利用率不足50%，严重制约并发能力。

### KV Cache 难以复用的问题

传统注意力机制中，KV Cache与具体请求强绑定，每个请求的KV Cache独立存储，即使多个请求存在相同的前缀（如多轮对话中，前几轮的对话内容相同），也无法复用已缓存的KV数据，只能重复计算并存储，导致内存冗余和计算浪费。例如，100个用户同时发起包含相同前缀的对话请求，传统方式会为每个请求单独存储一份前缀对应的KV Cache，重复占用100倍的内存空间，大幅增加显存压力。

## 10.2.2 PagedAttention 的核心设计思想

PagedAttention 的核心灵感来自操作系统的虚拟内存分页机制——将连续的逻辑内存空间（对应请求的KV Cache）切分为固定大小的块（Block），再将这些逻辑块映射到显存中的物理块，实现逻辑内存与物理内存的解耦，从而解决碎片和复用问题。其核心设计思想包括三点：

1. **借鉴操作系统虚拟内存分页机制**：类比操作系统中“虚拟地址-物理地址”的映射逻辑，PagedAttention 将每个请求的KV Cache（逻辑内存）视为一个“虚拟地址空间”，将GPU显存（物理内存）划分为固定大小的物理块，通过映射表管理逻辑块与物理块的对应关系，实现内存的动态分配和回收。

2. **KV Cache 切分为固定大小的 block（默认 16 个 token）**：vLLM 默认将每个请求的KV Cache按16个token为单位，切分为多个大小相等的逻辑块（Logical Block）。该块大小是经过性能权衡的——块太小会增加映射管理的开销，块太大则会导致内部碎片增加，16个token的块大小能在内存利用率和管理开销之间实现最优平衡。每个逻辑块对应一段连续的token序列，存储该序列对应的KV数据。

3. **逻辑 block 与物理 block 的动态映射**：每个请求的逻辑块不会固定绑定某块物理内存，而是由vLLM的内存管理器根据显存空闲情况，动态分配物理块，并通过映射表记录逻辑块与物理块的对应关系。当请求需要扩展上下文（如解码新token）时，仅需分配新的物理块并更新映射表；当请求结束时，释放其占用的所有物理块，这些物理块可立即被其他请求复用，从根本上解决外部碎片问题。同时，由于逻辑块大小固定，内部碎片仅为单个块内未使用的token空间（最多15个token），相比传统预分配方式，内部碎片大幅减少。

## 10.2.3 block table 与内存映射机制

block table（块表）是PagedAttention实现逻辑块与物理块映射的核心数据结构，本质上是一个二维数组，用于记录每个请求、每个Transformer层的KV Cache逻辑块对应的物理块地址。其核心设计和工作机制如下：

1. **block table 的结构**：block table 的维度为 [num_requests, num_layers, 2（K/V）, num_logical_blocks]，其中每个条目存储一个物理块的标识符（如物理块在显存中的起始地址）。对于每个请求，针对每一层的K和V，分别维护一个逻辑块到物理块的映射列表。

2. **内存映射的动态更新**：
        
    - 请求初始化（预填充阶段）：根据请求的初始上下文长度，计算所需的逻辑块数量，内存管理器从空闲物理块池中分配对应数量的物理块，更新block table，完成初始映射。
    
    - 解码阶段（生成新token）：每生成一个新token，若当前逻辑块已满（达到16个token），则分配一个新的物理块，将新token的KV数据写入该物理块，并更新block table，添加新的映射关系。
    
    - 请求结束：遍历该请求对应的block table条目，将所有占用的物理块归还给空闲物理块池，同时清空block table中的对应条目，实现物理块的复用。
    
3. **快速寻址与并行访问**：block table 支持快速查询，当注意力机制需要访问某一token的KV数据时，可通过token的位置计算出对应的逻辑块索引，再通过block table找到对应的物理块地址，直接访问显存中的数据。同时，由于每个物理块的地址独立，多线程可并行访问不同的物理块，提升注意力计算的并行效率。

通过block table的动态映射，vLLM实现了KV Cache的“按需分配、动态复用”，彻底解决了传统内存管理的碎片问题，大幅提升了显存利用率。

## 10.2.4 Copy-on-Write 与 KV Cache 共享

为进一步提升KV Cache的复用率，减少内存冗余，PagedAttention 引入了Copy-on-Write（写时复制）机制，支持多个请求共享相同前缀的KV Cache，仅在需要修改（写入新token）时才复制对应的物理块，从而节省大量显存空间。其核心工作原理如下：

1. **共享前缀的检测与映射复用**：当新请求发起时，vLLM会检测该请求的上下文前缀是否与已存在的请求（或缓存的前缀）一致。若一致，则直接复用已有的KV Cache物理块，在新请求的block table中，将对应逻辑块映射到已有的物理块，无需重新分配物理块和计算KV数据。

2. **写时复制的触发条件**：当共享物理块的某个请求需要写入新token（即解码阶段扩展上下文）时，若该物理块仍被其他请求共享，则触发Copy-on-Write机制——复制该物理块的内容到新的物理块，将当前请求的block table映射更新为新的物理块，然后在新物理块中写入新token的KV数据。而其他共享该物理块的请求，其映射关系保持不变，仍使用原物理块。

3. **优势与适用场景**：Copy-on-Write 机制尤其适用于多轮对话、批量推理等存在大量相同前缀的场景（如多个用户同时调用相同的系统提示词，或同一用户的多轮对话前缀相同）。通过共享KV Cache，可将内存占用降低数倍，同时减少重复计算，提升推理吞吐量。例如，100个共享相同前缀（1k token）的请求，通过Copy-on-Write仅需存储1份前缀KV Cache，相比传统方式节省99%的前缀内存占用。

## 10.2.5 PagedAttention 带来的性能提升

PagedAttention 通过解决传统内存管理的碎片和复用问题，为vLLM带来了显著的性能提升，核心体现在三个方面：

1. **显存利用率大幅提升**：通过动态分页和物理块复用，彻底解决了内部碎片和外部碎片问题，GPU显存利用率从传统方式的50%以下提升至80%以上，相同显存容量下可支持更多的并发请求。例如，在A100（40GB）显卡上，100亿参数模型（FP16）采用传统方式仅能支持8个4k上下文并发请求，而采用PagedAttention可支持15个以上，并发能力提升近一倍。

2. **吞吐量显著提升**：得益于更高的显存利用率和KV Cache共享，单位时间内可处理更多的推理请求，吞吐量相比传统推理框架（如Hugging Face Transformers）提升2-8倍，尤其在高并发场景下，性能优势更为明显。

3. **延迟优化**：虽然PagedAttention增加了少量映射管理开销，但通过减少内存分配时间、避免重计算（KV共享）、优化内存访问局部性，端到端延迟反而降低——首字延迟（TTFT）降低10%-30%，迭代延迟（ITL）基本持平或略有降低，实现了高吞吐量与低延迟的平衡。

# 10.3 内存压力下的动态调度

即使采用PagedAttention机制，当并发请求数量过多、上下文长度过长，或硬件显存不足时，仍会出现KV Cache空间不足的情况，导致新请求无法分配物理块，或已有的请求无法继续扩展上下文。为应对这种内存压力，vLLM引入了动态调度机制，核心是抢占（Preemption）机制，通过灵活的资源调度，平衡内存使用与推理性能，避免推理失败。

## 10.3.1 抢占（Preemption）机制

### 为什么需要抢占：KV 缓存空间不足

在高并发推理场景中，随着请求数量的增加，KV Cache占用的物理块数量会不断累积，当空闲物理块池耗尽时，新请求无法分配到足够的物理块，会被阻塞或拒绝；同时，若部分请求的上下文持续扩展（如长对话），会不断占用新的物理块，进一步挤压其他请求的内存空间。此时，需要通过抢占机制，回收部分已占用的物理块，分配给新请求或需要扩展的高优先级请求，确保推理过程的连续性和稳定性。

vLLM的抢占机制核心目标是：在内存不足时，通过最小化性能损失的方式，回收内存资源，优先保障高优先级请求的正常运行。

### V1 引擎中的抢占模式：默认使用 RECOMPUTE（重新计算）

vLLM V1引擎（当前主流引擎）支持多种抢占模式，默认采用RECOMPUTE（重新计算）模式，此外还有EVICT（驱逐）模式，两种模式的核心区别的在于内存回收的方式：

1. **RECOMPUTE（重新计算）模式**：这是vLLM的默认抢占模式。当需要回收某请求的物理块时，不会直接删除该请求的KV Cache数据，而是将该请求的部分逻辑块对应的物理块回收，分配给其他请求；当该请求后续需要访问被回收的KV数据时，通过重新计算的方式，重新生成对应的KV数据，并重新分配物理块。这种模式的优势是不会丢失请求的上下文信息，请求可以继续正常推理，仅在重新计算时会产生少量延迟开销，适合对请求连续性要求高的场景（如对话交互）。

 如果发生了抢占：会有如下信息：

```
WARNING 05-09 00:49:33 scheduler.py:1057 Sequence group 0 is preempted by PreemptionMode.RECOMPUTE mode because there is not enough KV cache space. This can affect the end-to-end performance. Increase gpu_memory_utilization or tensor_parallel_size to provide more KV cache memory. total_cumulative_preemption_cnt=1
```

vLLM会根据请求的优先级（可配置）和内存压力，自动选择需要抢占的请求和物理块数量，优先抢占低优先级、上下文较短或长时间未活跃的请求。

### 抢占对端到端延迟的影响

抢占机制虽然解决了内存不足的问题，但会对推理延迟产生一定影响，具体影响程度取决于抢占模式和抢占频率：

1. **RECOMPUTE模式的延迟影响**：当请求的KV数据被抢占后，后续解码时需要重新计算被回收的KV数据，会导致该请求的迭代延迟（ITL）暂时增加（增加幅度取决于重新计算的token数量和模型层数）。但由于仅重新计算被抢占的部分，而非全部KV数据，延迟增加通常在可接受范围内（一般增加10%-50%），且仅在抢占后的第一次解码时出现，后续解码恢复正常。

总体而言，合理的抢占策略可以在内存压力下，以轻微的延迟代价，保障整体推理的稳定性和高吞吐量，避免因内存不足导致的推理失败。

## 10.3.2 监控抢占：Prometheus 指标与日志

为了实时掌握抢占机制的运行状态，优化抢占策略，vLLM集成了Prometheus监控指标，并在日志中记录抢占相关信息，方便开发者监控和排查问题。

### Prometheus 监控指标

vLLM暴露的与抢占相关的核心Prometheus指标如下，可通过Grafana等工具可视化展示：

1. `vllm_preemption_count`：累计抢占次数，按抢占模式（recompute/evict）和请求优先级分类，用于衡量抢占的频率。

2. `vllm_preemption_recompute_time_seconds`：累计重新计算时间，用于衡量RECOMPUTE模式的延迟开销。

3. `vllm_physical_blocks_used`：当前已使用的物理块数量，用于判断内存压力。

4. `vllm_physical_blocks_free`：当前空闲的物理块数量，用于判断是否需要触发抢占。

5. `vllm_requests_preempted_total`：累计被抢占的请求数量，按请求优先级分类。

### 日志监控

vLLM的日志中会详细记录抢占相关事件，包括：抢占触发时间、被抢占的请求ID、抢占模式、回收的物理块数量、重新计算的token数量等。通过日志可以定位具体的抢占场景，分析抢占原因（如是否因并发过高、上下文过长导致），为调优提供依据。例如，日志中会出现类似如下条目：`[INFO] Preempting request 123 (priority: low) using RECOMPUTE mode, reclaiming 8 physical blocks.`

## 10.3.3 减少抢占的调优策略

频繁的抢占会增加延迟开销，影响推理性能，因此在实际部署中，应尽量减少抢占的发生。核心调优策略围绕“降低内存压力”展开，具体如下：

1. **增加 gpu_memory_utilization**：这是vLLM的核心配置参数之一，用于设置GPU显存的利用率阈值（默认0.9，即90%）。适当提高该值（如0.95），可以让vLLM更充分地利用GPU显存，减少因显存未充分利用导致的空闲物理块不足，从而减少抢占。但需注意，该值不宜设置过高（如超过0.98），否则可能导致显存溢出（OOM）。

2. **减少 max_num_seqs 或 max_num_batched_tokens**：
        这两个参数的调整需要结合业务场景，在并发量和吞吐量之间找到平衡。

    - `max_num_seqs`：最大并发请求数，减少该值可以降低同时运行的请求数量，从而减少KV Cache的总内存占用，缓解内存压力，减少抢占。

    - `max_num_batched_tokens`：单批次最大token数量，减少该值可以降低每个批次的内存开销，避免因单批次内存占用过高导致的空闲物理块不足。

3. **增加 tensor_parallel_size 或 pipeline_parallel_size**：
        该策略适用于多GPU部署场景，需要根据GPU数量和模型规模合理配置。

    - `tensor_parallel_size`：张量并行度，增加该值可以将模型参数和KV Cache分散到多个GPU上，每个GPU仅承担部分内存压力，从而降低单GPU的内存占用，减少抢占。例如，将tensor_parallel_size从1调整为2，单GPU的参数和KV Cache内存占用可降低约50%。

    - `pipeline_parallel_size`：流水线并行度，增加该值可以将模型的不同层分散到多个GPU上，减少单GPU的中间激活值和KV Cache占用，缓解内存压力。

# 10.4 分块预填充（Chunked Prefill）：平衡计算与内存

预填充（Prefill）是大模型推理的第一个阶段，指将用户输入的prompt（提示词）输入模型，计算并缓存所有token的KV数据，为后续的解码阶段做准备。传统预填充采用“一次性计算”的方式，即对整个prompt的token序列进行一次性计算，这种方式在prompt较长时，会导致中间激活值大幅累积，占用大量GPU显存，甚至引发OOM；同时，一次性计算会导致GPU计算资源瞬时饱和，影响其他请求的处理。为解决这一问题，vLLM引入了分块预填充（Chunked Prefill）机制，将预填充分为多个小块，与解码请求混合批处理，实现计算与内存的平衡。

## 10.4.1 传统预填充 vs. 分块预填充

传统预填充与分块预填充的核心差异的在于预填充阶段的计算方式和内存占用，具体对比如下：

1. **传统预填充**：
        

    - 计算方式：对整个prompt的token序列（假设长度为L）进行一次性计算，生成所有token的KV Cache，中间激活值在计算过程中一次性累积。

    - 内存开销：中间激活值的内存占用与L的平方成正比（因为Transformer的注意力计算需要处理所有token对），当L较大（如4k、8k）时，中间激活值会占用数十GB的显存，导致显存压力骤增。

    - 性能问题：一次性计算会导致GPU计算资源瞬时负载过高，出现“计算峰值”，同时阻塞其他请求的解码计算，影响整体吞吐量；若L过大，可能直接导致OOM。

2. **分块预填充**：
        

    - 计算方式：将prompt的token序列切分为多个固定大小的chunk（块），每个chunk的大小通常为`max_num_batched_tokens`的一部分，逐块进行预填充计算，每计算完一个chunk，就释放该chunk的中间激活值，同时缓存该chunk的KV Cache。

    - 内存开销：中间激活值的内存占用仅与单个chunk的大小成正比，而非整个prompt的长度，大幅降低了预填充阶段的显存压力。例如，将8k长度的prompt切分为4个2k的chunk，中间激活值的内存占用可降低至传统方式的1/4。

    - 性能优势：逐块计算避免了计算峰值，GPU资源可均匀分配；同时，预填充的chunk计算可与其他请求的解码计算混合批处理，提升GPU利用率和整体吞吐量。

总结而言，分块预填充通过“分而治之”的方式，将预填充阶段的内存压力分散，同时优化计算调度，实现了预填充阶段计算效率与内存开销的平衡，尤其适合长prompt推理场景。

## 10.4.2 工作原理：将预填充分块并与解码请求混合批处理

分块预填充的核心工作原理分为两个步骤：分块计算和混合批处理，具体如下：

1. **预填充分块**：当收到一个长prompt请求时，vLLM会根据`max_num_batched_tokens`和prompt长度，自动将prompt切分为多个chunk。例如，若`max_num_batched_tokens`设置为4096，prompt长度为8192，则会切分为2个4096 token的chunk。每个chunk的大小可通过参数微调，但通常不超过`max_num_batched_tokens`的一半，以预留空间给解码请求。

2. **混合批处理**：vLLM的调度器会将预填充的chunk计算与其他请求的解码计算（每个解码请求仅生成1个token，内存开销小）混合在一起，组成一个批次进行计算。例如，当一个预填充chunk（4096 token）正在计算时，调度器会同时加入多个解码请求（每个1 token），充分利用GPU的计算资源，避免预填充计算独占GPU。

3. **KV Cache 拼接**：每个chunk计算完成后，其对应的KV Cache会被存储到物理块中，通过block table记录映射关系；当所有chunk计算完成后，vLLM会将各个chunk的KV Cache逻辑块拼接起来，形成完整的prompt KV Cache，用于后续的解码阶段。

这种机制的核心优势是：既降低了预填充阶段的内存压力，又提升了GPU利用率，避免了预填充与解码之间的资源竞争，实现了整体推理性能的优化。

## 10.4.3 调度策略：优先解码，利用 max_num_batched_tokens 预算

分块预填充的调度策略核心是“优先解码，合理分配批处理预算”，确保解码请求的低延迟，同时充分利用`max_num_batched_tokens`的内存和计算预算，具体规则如下：

1. **解码优先原则**：vLLM的调度器会优先处理解码请求，因为解码请求的延迟要求更高（尤其是实时交互场景），且每个解码请求仅占用1个token的批处理预算，内存和计算开销小。只有当GPU有空闲计算资源，且`max_num_batched_tokens`有剩余预算时，才会调度预填充chunk的计算。

2. **max_num_batched_tokens 预算分配**：`max_num_batched_tokens`是单批次最大token数量，用于限制该批次的总内存和计算开销。调度器会为预填充chunk分配一部分预算（通常为50%-70%），剩余预算留给解码请求。例如，若`max_num_batched_tokens`=4096，为预填充分配2048 token的预算，剩余2048 token用于解码请求，可同时处理2048个解码请求（每个1 token）。

3. **动态调整**：调度器会根据当前的请求队列（预填充请求数量、解码请求数量）和内存压力，动态调整预填充chunk的大小和批处理预算分配。当解码请求较多时，减少预填充的预算占比；当预填充请求较多时，适当增加预填充的预算占比，确保整体性能平衡。

## 10.4.4 性能调优：max_num_batched_tokens 的选择策略

`max_num_batched_tokens`是分块预填充的核心调优参数，其取值直接影响预填充的内存开销、计算效率和整体吞吐量，需根据业务场景（延迟要求、prompt长度、并发量）合理选择，具体策略如下：

1. **较小值（如 2048）：改善 ITL（迭代延迟）**：
        

    - 适用场景：低延迟交互场景（如对话机器人），对迭代延迟（ITL）要求较高，prompt长度较短（通常小于2k）。

    - 优势：单批次token数量少，GPU计算压力小，解码请求的响应速度快，ITL显著改善；同时，预填充chunk的大小较小，中间激活值内存占用低，不易引发OOM。

    - 劣势：批次数量增加，批次切换开销略有增加，整体吞吐量可能略有下降。

2. **较大值（>8192）：改善 TTFT（首字延迟）和吞吐量**：


    - 适用场景：高吞吐量批量推理场景（如文本生成、长文档分析），对首字延迟（TTFT）和吞吐量要求较高，prompt长度较长（通常大于4k）。
    
    - 优势：单批次token数量多，可同时处理更多的预填充chunk和解码请求，批次切换开销减少，整体吞吐量大幅提升；同时，预填充chunk的大小较大，减少了chunk数量，降低了KV Cache拼接的开销，TTFT（首字延迟）改善明显。
    
    - 劣势：单批次内存开销大，中间激活值占用较高，需要更大的GPU显存；若并发量过高，可能导致内存压力增加，引发抢占。

经验值：对于大多数场景，建议将`max_num_batched_tokens`设置为4096-8192，兼顾ITL、TTFT和吞吐量；若GPU显存充足（如A100 80GB），且处理长prompt批量推理，可设置为16384；若显存有限，且追求低延迟，可设置为2048。

## 10.4.5 与 max_model_len 的关系及注意事项

`max_model_len`是模型支持的最大上下文长度，分块预填充的实现与`max_model_len`密切相关，同时存在一些需要注意的细节，具体如下：

1. **与 max_model_len 的关系**：
        

    - `max_num_batched_tokens`的取值不能超过`max_model_len`，否则会导致批次token数量超出模型的最大上下文限制，引发推理错误。

    - 当prompt长度接近或等于`max_model_len`时，分块预填充的优势更为明显——传统预填充会因中间激活值过大导致OOM，而分块预填充可通过切分chunk，将中间激活值控制在可接受范围内。

    - 若`max_model_len`设置过大（如32k），而`max_num_batched_tokens`设置过小（如2048），会导致预填充chunk数量过多，KV Cache拼接开销增加，影响TTFT，此时需适当增大`max_num_batched_tokens`。

2. **注意事项**：
        

    - 分块预填充仅适用于预填充阶段，解码阶段仍采用单token计算，不受`max_num_batched_tokens`的直接限制。

    - 当启用分块预填充时，需确保GPU显存有足够的空闲空间，用于存储每个chunk的中间激活值和KV Cache，避免因chunk计算导致的内存溢出。

    - 对于极短prompt（如小于128 token），分块预填充的优势不明显，反而可能增加调度开销，此时vLLM会自动禁用分块预填充，采用传统一次性预填充。

# 10.5 跨层级内存扩展：KV 缓存卸载（Offloading）

即使采用PagedAttention和分块预填充机制，当处理超大并发请求或超长上下文时，GPU显存仍可能不足。为进一步扩展内存容量，vLLM引入了KV缓存卸载（KV Offloading）机制，将部分KV Cache从GPU显存卸载到CPU内存（或外部存储），实现跨层级内存扩展，在不增加GPU硬件成本的前提下，提升模型的并发能力和上下文支持能力。

## 10.5.1 为什么要卸载？

KV缓存卸载的核心目的是缓解GPU显存压力，解决高并发、长上下文场景下的内存不足问题，具体原因有两点：

1. **处理大量并发请求时的 GPU 内存不足**：当并发请求数量超过GPU显存的承载能力时，即使采用PagedAttention和抢占机制，也无法避免内存压力过大，导致请求被阻塞或OOM。此时，将部分KV Cache卸载到CPU内存（容量通常远大于GPU显存，如服务器CPU内存可达128GB-1TB），可释放GPU显存，容纳更多的并发请求。

2. **避免被抢占请求的重新计算成本**：在内存压力下，RECOMPUTE模式的抢占会导致请求重新计算KV数据，增加延迟开销；而通过KV缓存卸载，将不常用的KV Cache（如低优先级请求的KV数据）卸载到CPU内存，可减少抢占的发生，从而避免重新计算的成本，提升推理性能。

需要注意的是，KV缓存卸载仅针对KV Cache，模型参数权重仍需驻留GPU显存（除非启用模型参数卸载，但会大幅增加延迟），因此卸载的核心是释放GPU显存中KV Cache占用的空间。

## 10.5.2 CPU 卸载的优势

在众多卸载目标（CPU内存、外部存储如SSD）中，CPU内存是最常用的选择，其核心优势体现在三点：

1. **CPU RAM 广泛可用且容量大**：几乎所有服务器都配备大容量CPU内存，且容量远高于GPU显存（如普通服务器CPU内存为64GB-128GB，高端服务器可达512GB-1TB），可提供充足的内存空间用于存储卸载的KV Cache，满足高并发、长上下文的需求。

2. **GPU-CPU 传输的低延迟和高吞吐量**：通过PCIe总线，GPU与CPU之间的内存传输延迟较低（通常为几十微秒），吞吐量较高（PCIe 4.0可达32GB/s），相比外部存储（如SSD，传输延迟为毫秒级），可大幅降低KV Cache加载/存储的延迟开销，减少对推理性能的影响。

3. **作为进一步卸载到外部存储的中转站**：当CPU内存也不足时，可将CPU内存中的部分KV Cache进一步卸载到外部存储（如SSD、NVMe），此时CPU内存作为中转站，负责GPU与外部存储之间的KV数据传输，优化传输效率，避免直接卸载到外部存储导致的高延迟。

## 10.5.3 卸载连接器（Offloading Connector）架构

vLLM通过卸载连接器（Offloading Connector）架构实现KV缓存的卸载与管理，该架构采用模块化设计，支持多种卸载后端（CPU、SSD等），核心组件和设计特点如下：

1. **异步加载/存储 API**：卸载连接器提供异步的KV数据加载和存储API，允许GPU在进行推理计算的同时，并行执行KV数据的CPU-GPU传输，避免传输过程阻塞推理计算，降低延迟开销。例如，当GPU正在解码某请求的token时，卸载连接器可异步将该请求后续需要的KV数据从CPU内存加载到GPU显存，或异步将不常用的KV数据从GPU显存存储到CPU内存。

2. **可插拔的后端设计**：卸载连接器采用可插拔架构，支持不同的卸载后端，开发者可根据业务需求和硬件环境选择合适的后端，无需修改核心代码。目前vLLM支持的后端包括：CPU原生后端（默认）、SSD后端（实验性）、分布式内存后端（适用于多节点部署）。

3. **CPU 后端的原生实现**：CPU后端是vLLM的默认卸载后端，采用高效的内存管理策略，将卸载的KV Cache存储在CPU内存的连续缓冲区中，通过内存映射（mmap）机制优化数据访问效率，减少CPU-GPU传输的开销。同时，CPU后端支持动态分配缓冲区大小，根据KV Cache的卸载量自动调整，避免CPU内存浪费。

## 10.5.4 启用 CPU 卸载的方法

vLLM启用CPU卸载的配置方式简单，主要通过命令行参数设置，支持两种配置方式（新版本和旧版本）：

1. **新版本配置方式（推荐）**：通过`--kv_offloading_backend`指定卸载后端，通过`--kv_offloading_size`指定卸载到CPU内存的最大容量（单位：GB）。例如：
        `vllm serve model_name --kv_offloading_backend native --kv_offloading_size 32`该命令表示启用CPU原生卸载后端，最大卸载32GB的KV Cache到CPU内存。

2. **旧版本的配置方式**：旧版本vLLM通过`--cpu_offload`参数启用CPU卸载，通过`--cpu_offload_size`指定卸载容量。例如：
        `vllm serve model_name --cpu_offload --cpu_offload_size 32`注意：旧版本的配置方式已被逐步废弃，建议使用新版本的`--kv_offloading_backend`参数。

补充说明：`--kv_offloading_size`的取值应根据CPU内存容量和业务需求合理设置，建议不超过CPU内存的50%，避免占用过多CPU内存影响其他进程的运行。

## 10.5.5 卸载性能分析

KV缓存卸载虽然能缓解GPU内存压力，但会因CPU-GPU数据传输产生一定的延迟开销，整体性能受缓存命中率影响较大，具体性能表现如下：

1. **TTFT 降低 2-22 倍（从 CPU 加载）**：首字延迟（TTFT）的增加主要来自于预填充阶段KV数据的加载——若预填充的KV数据需要从CPU内存加载到GPU显存，会增加数据传输延迟，导致TTFT比无卸载时降低2-22倍（具体取决于传输数据量和PCIe带宽）。但当KV数据已缓存到GPU显存（缓存命中）时，TTFT与无卸载时基本一致。

2. **吞吐量提升最高 9 倍（高缓存命中率）**：在高并发场景下，当缓存命中率较高（如超过80%）时，KV缓存卸载可释放大量GPU显存，支持更多的并发请求，整体吞吐量相比无卸载时提升最高9倍；若缓存命中率较低（如低于50%），则会因频繁的CPU-GPU传输导致延迟增加，吞吐量提升不明显，甚至可能下降。

核心结论：KV缓存卸载的性能关键在于缓存命中率，建议在高并发、请求上下文重复度高（如多轮对话）的场景下使用，可获得最佳的性能收益。
