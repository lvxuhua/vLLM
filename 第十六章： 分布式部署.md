vLLM 分布式部署的核心目标是突破单卡显存与算力限制，支持超大模型（如 13B、70B）推理与高吞吐服务，适配大规模生产场景。其并行体系基于 Megatron-LM 算法实现，默认采用「单节点多进程、多节点 Ray」的执行后端，可灵活组合多种并行策略，兼顾性能与兼容性。本章严格遵循官方规范，按并行策略分类讲解，配套完整实操步骤，适配教学落地。

## 16.1 张量并行（Tensor Parallelism）

张量并行是 vLLM 最常用的分布式并行方式，核心适配单节点多卡场景，通过切分模型层参数与计算任务，实现多卡协同推理，解决单卡显存不足、算力不足的问题，无需额外依赖分布式后端（单节点默认使用多进程后端）。

### 16.1.1 核心原理

将模型的**层参数与计算任务**按张量维度切分到多块 GPU 上，同一层的不同 GPU 并行执行计算，通过 All-Reduce 通信操作同步梯度、中间结果，最终汇总输出。简单来说，就是“把一个大模型的层，拆分成多份，让多卡一起算同一层”，核心解决单卡显存无法容纳大模型的问题。

核心特点：单节点多卡协同，通信开销低（仅单节点内通信），并行效率高，适合 7B~70B 模型的单节点部署。

### 16.1.2 核心配置

通过 `--tensor-parallel-size <N>` 参数指定张量并行度，即单节点内参与并行的 GPU 数量，核心要求：

- N 需为正整数，且 ≤ 当前节点的 GPU 总数；

- 优先选择 2 的幂次（2、4、8），官方推荐该配置，通信效率最优；

- 无需额外配置后端，单节点场景默认使用 `mp`（多进程）后端，自动分配 GPU 资源。

### 16.1.3 实操步骤

### 16.1.4 注意事项

- 单节点内建议使用 NVLink/SLI 连接 GPU，若无则需确保 PCIe 带宽充足，避免通信瓶颈；

- 并行度 N 不可超过模型支持的最大并行度（如 7B 模型通常支持最大 8 卡并行）；

- 若出现 GPU 显存分配不均，需检查是否关闭其他占用 GPU 资源的进程，或降低 `--max_num_seqs` 参数。

## 16.2 流水线并行（Pipeline Parallelism）

流水线并行核心适配多节点场景，当模型过大（如 70B 及以上），单节点多卡仍无法容纳时，通过按层切分模型，让不同节点的 GPU 负责不同阶段的计算，实现多节点接力推理，突破单节点资源限制。

### 16.2.1 核心原理

将模型**按层切分**为多个连续的阶段（如模型共 40 层，切分为 2 个阶段，每个阶段 20 层），不同节点的 GPU 负责一个阶段的计算，推理请求按“阶段1→阶段2→...→输出”的顺序接力处理，通过激活值传递实现流水线执行。简单来说，就是“把一个大模型的层，按顺序拆分成多段，让多节点的 GPU 依次处理每一段”。

核心特点：多节点协同，突破单节点显存限制，适合超大模型部署；通信开销高于张量并行（跨节点通信），需依赖高速网络。

### 16.2.2 核心配置
通过 `--pipeline-parallel-size <M>` 参数指定流水线并行度，即参与并行的节点数量，核心配置规则：

- M 需为正整数，且 = 参与部署的节点总数；

- 通常与张量并行结合使用：`tensor-parallel-size` 为「每节点 GPU 数」，`pipeline-parallel-size` 为「节点数」；

- 多节点场景需指定分布式后端，推荐使用 Ray（官方默认）。

## 16.3 专家并行（Expert Parallelism）

专家并行是针对 MoE（Mixture of Experts，混合专家模型）的专属分布式优化方案，MoE 模型包含多个“专家层”，通过将不同专家层分配到不同 GPU 上，实现专家计算并行，同时优化负载均衡与通信效率，提升 MoE 模型推理吞吐量。

### 16.3.1 核心原理
MoE 模型的核心是“多个专家层+门控网络”，门控网络决定每个输入样本由哪些专家处理。专家并行将这些**专家层切分到不同 GPU** 上，每个 GPU 负责部分专家的计算，门控网络根据输入样本分配专家，通过 All-to-All 通信实现样本与专家的匹配，最终汇总计算结果。

核心特点：仅适配 MoE 模型（如 Mixtral-8x7B），不支持普通 dense 模型；可与张量并行、流水线并行组合使用，大幅提升 MoE 模型推理效率。

### 16.3.2 核心优化点（
1. **负载均衡优化**vLLM 内置专家负载均衡策略，通过动态调整样本分配，避免部分 GPU 专家过载、部分 GPU 空闲，确保所有 GPU 利用率趋于均衡。核心机制：门控网络输出专家分配概率时，加入负载均衡因子，优先分配样本到空闲专家。

2. **All-to-All 通信优化**专家并行的核心通信是 All-to-All（全对全）通信（样本分配到对应专家 GPU、专家计算结果汇总），vLLM 优化了通信逻辑，支持根据 GPU 数量、专家数量动态调整通信粒度，降低通信延迟，提升并行效率。

### 16.3.4 注意事项

- 仅适配 MoE 模型，普通 dense 模型开启专家并行无效，会报错；

- 专家数量需与并行度匹配，建议专家数量是 GPU 数量的整数倍，提升负载均衡效果；

- 多节点专家并行需依赖 Ray 后端，且需确保节点间通信带宽充足，避免 All-to-All 通信瓶颈。

### 16.3.5 最佳实践

- 多机部署优先使用 Ray 后端，兼容性最好，官方持续维护，不推荐其他后端；

- 模型文件建议使用共享存储（如 NFS），避免在每个节点单独复制，节省存储空间；

- 大规模部署（≥4 节点）建议开启 Ray 可视化面板（dashboard-port），便于监控节点状态与资源使用；

- 跨节点通信优先使用 InfiniBand，若使用以太网，建议开启 RDMA 加速，降低通信延迟。
