# 第13章 性能调优实战

性能调优是 vLLM 部署落地的核心环节，核心目标是在保证推理精度的前提下，最大化提升推理吞吐量、降低延迟，适配高并发、低延迟的业务场景。本章聚焦实战，从性能指标解读、负载测试方法、核心参数调优三个维度，结合官方实操案例，完整讲解 vLLM v0.7.0 性能调优全流程，助力高效落地部署。

## 13.1 性能指标解读：吞吐量（tokens/s）、首字延迟（TTFT）、每个输出 token 时间（TPOT）

vLLM 官方定义的三大核心性能指标，是衡量推理性能的关键，需精准理解各指标含义、计算逻辑及优化目标，为后续调优提供依据，所有指标均基于 vLLM 官方日志输出，可直接读取分析。

### 13.1.1 核心指标详解（官方定义）

1. **吞吐量（Throughput，单位：tokens/s）**定义：单位时间内（每秒）模型生成的 token 总数量，是衡量模型并发处理能力的核心指标，数值越高，并发性能越强，适配高流量场景。计算逻辑：吞吐量 = 总生成 token 数量 ÷ 推理总耗时（秒），vLLM 日志中直接输出「Throughput: X tokens/s」，无需手动计算。优化目标：在保证延迟可控的前提下，最大化提升吞吐量；官方参考标准：7B 模型单机部署，吞吐量≥50 tokens/s（高并发场景）。

2. **首字延迟（Time to First Token，TTFT，单位：毫秒 ms）**定义：从用户发送推理请求，到模型生成第一个 token 的时间，是衡量“响应速度”的核心指标，数值越低，用户体验越好，适配低延迟场景（如实时对话）。核心影响因素：模型加载速度、CPU 分词（tokenization）速度、GPU 内核启动开销，是短序列推理（如问答）的核心优化靶点。优化目标：TTFT ≤ 500ms（实时对话场景）、TTFT ≤ 1000ms（批量生成场景）；官方参考：7B 模型单机部署，默认配置下 TTFT 约 300~800ms。

3. **每个输出 token 时间（Time per Output Token，TPOT，单位：毫秒 ms）**定义：生成第一个 token 后，每生成一个后续 token 所需的平均时间，是衡量“持续生成速度”的核心指标，数值越低，长序列生成效率越高。计算逻辑：TPOT = （推理总耗时 - TTFT）÷（总生成 token 数量 - 1），vLLM 日志中可间接计算，部分版本支持直接输出。优化目标：TPOT ≤ 50ms（长序列生成场景）；官方参考：7B 模型单机部署，默认配置下 TPOT 约 20~40ms。

### 13.1.2 指标关联与优化优先级

三大指标并非孤立，存在一定权衡关系，调优时需根据业务场景优先排序：

- 实时对话场景（如客服机器人）：优先优化 TTFT（低延迟），其次优化 TPOT，最后兼顾吞吐量；

- 批量生成场景（如文本续写、数据标注）：优先优化吞吐量（高并发），其次优化 TPOT，对 TTFT 要求较低；

- 通用场景：平衡三大指标，确保 TTFT ≤ 1000ms、TPOT ≤ 50ms、吞吐量≥30 tokens/s（7B 模型）。

## 13.2 负载测试方法论

负载测试是性能调优的前提，核心是通过模拟不同并发场景，测试模型在不同负载下的性能表现，定位性能瓶颈，为参数调优提供数据支撑。vLLM 官方推荐使用 locust、wrk 或自定义脚本进行压测，重点分析不同并发度下的性能曲线。

### 13.2.2 不同并发度下的性能曲线分析

负载测试的核心是分析“并发度 vs 性能指标”的关系，通过绘制性能曲线，定位最优并发度（避免资源浪费或性能瓶颈），官方推荐的分析流程如下：

1. **测试准备**固定模型（如 Qwen3-8B）、固定 prompt 长度（如 50 token）、固定输出 token 数量（如 100 token），仅调整并发度（从 1 逐步增加到 50），每个并发度测试 3 次，取平均值。

2. **核心性能曲线解读**通过 Locust 可视化页面或自定义脚本输出数据，绘制 3 条核心曲线，重点分析拐点：

    - 并发度 vs 吞吐量曲线：并发度从 1 增加时，吞吐量线性上升；当并发度达到“最优值”后，吞吐量趋于平稳甚至下降（GPU 资源饱和）；

    - 并发度 vs TTFT 曲线：并发度从 1 增加时，TTFT 缓慢上升；当并发度超过最优值后，TTFT 急剧上升（CPU/GPU 资源竞争加剧）；

    - 并发度 vs TPOT 曲线：并发度较低时，TPOT 基本稳定；当并发度超过最优值后，TPOT 逐渐上升（GPU 计算压力增大）。

3. **最优并发度定位（官方示例）**以 7B 模型、16GB 显存 GPU 为例，官方测试的最优并发度的参考范围：10~20；超过 20 后，吞吐量趋于平稳，TTFT 急剧上升，此时无需继续增加并发度。实操结论：不同模型（7B/13B）、不同显存大小，最优并发度不同，需通过负载测试自行定位，核心原则是“吞吐量最大化，同时 TTFT/TPOT 可控”。

## 13.3 配置参数调优指南

vLLM  的性能调优核心是“参数适配硬件资源”，重点围绕 max_num_seqs、max_model_len 两大核心参数，结合上下文长度、CPU 线程数等辅助参数，实现性能最大化，所有参数均通过 CLI 配置.

### 13.3.1 max_num_seqs、max_model_len 的平衡（核心调优参数）

两个参数直接决定 GPU 内存占用和并发处理能力，需根据 GPU 显存大小、业务场景（上下文长度）灵活调整，核心原则是“不超出显存，同时最大化并发”。

1. **参数解析（官方定义）**

    - `--max_num_seqs <int>`：单卡最大并发序列数，即单卡同时处理的推理请求数量（默认值：256）；数值越大，并发能力越强，但内存占用越高，易导致 OOM（内存溢出）。

    - `--max_model_len <int>`：模型支持的最大上下文长度（默认值：2048），即 prompt + 输出 token 的最大总长度；数值越大，支持的长序列越多，但内存占用越高，吞吐量会略有下降。

### 13.3.3 调优总结

vLLM 性能调优的核心是“硬件资源适配+业务场景匹配”，步骤总结如下：

1. 通过负载测试，定位最优并发度（绘制性能曲线）；

2. 根据 GPU 显存，调整 max_num_seqs、max_model_len，平衡并发与内存；

3. 根据上下文长度，权衡吞吐量与延迟，调整对应参数；

4. 优化 CPU 线程数，降低分词时间，减少 TTFT；

5. 通过 vLLM 日志（--verbose）监控性能指标，迭代优化，直至满足业务需求。
