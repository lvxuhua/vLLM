# 第14章 服务化部署进阶

服务化部署进阶是 vLLM 从测试环境走向生产环境的关键环节，核心目标是实现 API 服务的高可用、高安全、可扩展，适配大规模、多场景的生产部署需求。本章聚焦服务化部署的高级配置，从 API 高级配置、服务发现与负载均衡、优雅启停、多模型资源隔离四个维度，结合官方实操案例，完整讲解 vLLM 服务化部署进阶全流程，助力生产环境稳定落地。

## 14.1 API 服务高级配置

vLLM 的 API 服务默认提供基础推理接口，生产环境需补充请求限流、HTTPS 认证、CORS 跨域等高级配置，保障服务安全、稳定运行，所有配置均通过 CLI 参数或配置文件实现，官方推荐 CLI 配置（实操便捷，适配教学演示）。

### 14.1.1 请求限流

```
核心目的：防止高并发请求压垮服务，避免单个慢请求占用过多资源，保障服务整体可用性。vLLM 支持通过参数直接配置限流和超时，无需额外依赖第三方工具。
1. **核心参数解析（官方定义）**
   - `--max-num-batched-tokens <int>`：单批次最大 token 数量，间接实现请求限流（默认值：4096）；数值越小，限流越严格，避免单批次请求占用过多 GPU 资源。
   - --max-num-seqs: 最大并发数
```

## 14.2 服务发现与负载均衡

当 vLLM 服务部署多实例（多 GPU 单机或多机集群）时，需实现服务发现与负载均衡，确保请求均匀分配到各个实例，提升服务可用性和并发处理能力。vLLM 本身不提供服务发现功能，官方推荐结合 Nginx、Consul 或 Kubernetes 实现，以下重点讲解最常用的 Nginx 负载均衡配置（实操简单，适配教学演示）。

### 14.2.1 部署架构（官方推荐）

核心架构：前端请求 → Nginx（负载均衡+反向代理）→ 多 vLLM 实例（后端服务），其中多 vLLM 实例需确保模型一致、端口不同，部署在同一局域网或可互通的服务器上。

示例部署：2 个 vLLM 实例，分别部署在本机的 8001 和 8002 端口，Nginx 监听 443 端口（HTTPS），实现请求分发。

### 14.2.2 多 vLLM 实例启动（实操）

## 14.3 优雅启动与关闭

优雅启动与关闭是保障服务稳定性的重要环节：优雅启动确保服务完全就绪后再接收请求，避免请求失败；优雅关闭确保正在处理的请求完成后再终止服务，避免数据丢失或请求中断，vLLM 支持通过参数和命令实现优雅启停。

### 14.3.1 优雅启动
vLLM 启动时，默认会先加载模型到 GPU，加载完成后才启动 API 服务，无需额外配置；若需延迟接收请求（如等待依赖服务就绪），可结合 shell 脚本实现。

```bash
# 优雅启动脚本（等待模型加载完成，延迟 10 秒接收请求）
#!/bin/bash
# 启动 vLLM 服务，后台运行
nohup vllm serve /mnt/moark-models/Qwen3-8B \
--trust-remote-code \
--dtype float16 \
--port 8000 \
--host 0.0.0.0 

# 记录进程 ID
PID=$!

# 等待模型加载完成（检测日志中的 "Started server on" 关键字）
echo "等待 vLLM 模型加载完成..."
while ! grep -q "Started server on" vllm.log; do
    sleep 2
done

# 延迟 10 秒，确保服务完全就绪
sleep 10
echo "vLLM 服务已优雅启动，进程 ID：$PID"

```

### 14.3.2 优雅关闭
vLLM 支持接收 SIGTERM 信号（优雅关闭信号），收到信号后，会停止接收新请求，等待正在处理的请求完成后，再终止进程，避免请求中断。

```bash
# 1. 查找 vLLM 进程 ID
PID=$(ps aux | grep "vllm serve" | grep -v grep | awk '{print $2}')

# 2. 发送 SIGTERM 信号，优雅关闭（推荐）
kill -SIGTERM $PID

# 3. 等待进程终止（可选，查看日志确认）
echo "正在优雅关闭 vLLM 服务..."
while ps -p $PID > /dev/null; do
    sleep 1
done
echo "vLLM 服务已优雅关闭"

# 说明：若发送 SIGTERM 后进程未终止，可发送 SIGKILL 强制关闭（不推荐，可能导致请求中断）
# kill -SIGKILL $PID

```

### 14.3.3 系统服务配置
将 vLLM 配置为系统服务，实现开机自启、优雅启动和关闭，适配生产环境长期运行。

```bash
# 1. 创建系统服务文件
cat > /etc/systemd/system/vllm.service << EOF
[Unit]
Description=vLLM API Service
After=network.target nvidia-smi.service  # 依赖网络和 NVIDIA 驱动

[Service]
User=root  # 替换为实际用户
WorkingDirectory=/root/vllm  # 替换为 vLLM 运行目录
ExecStart=/usr/bin/python3 -m vllm.serve.openai_api_server \
  --model hermes-2-pro:7b \
  --trust-remote-code \
  --dtype float16 \
  --port 8000 \
  --host 0.0.0.0 \
  --max_num_seqs 15 \
  --verbose
ExecStop=/bin/kill -SIGTERM \$MAINPID  # 优雅关闭
Restart=on-failure  # 服务失败时自动重启
RestartSec=5  # 重启间隔 5 秒

[Install]
WantedBy=multi-user.target
EOF

# 2. 启用并启动服务
systemctl daemon-reload
systemctl enable vllm  # 开机自启
systemctl start vllm   # 启动服务

# 3. 查看服务状态、停止服务
systemctl status vllm  # 查看状态
systemctl stop vllm    # 优雅停止服务

```

## 14.4 多模型服务的资源隔离

当需要在同一台服务器（多 GPU）或同一集群中部署多个 vLLM 模型时，需实现资源隔离，避免不同模型之间抢占 GPU、CPU、内存资源，确保每个模型服务稳定运行。vLLM 结合 NVIDIA GPU 调度和系统资源限制，实现多模型资源隔离，官方推荐两种方案。

### 14.4.1 方案1：单服务器多 GPU 隔离（单机多模型）

核心逻辑：为每个模型分配独立的 GPU，通过 `--gpu-ids` 参数指定模型使用的 GPU 编号，实现 GPU 资源完全隔离；同时通过 `--cpu-cores` 限制每个模型使用的 CPU 核心数，避免 CPU 资源竞争。

```bash
# 示例：单机 2 个 GPU（编号 0 和 1），分别部署 7B 和 13B 模型，实现资源隔离
# 1. 部署模型 1（hermes-2-pro:7b），使用 GPU 0，限制 8 个 CPU 核心
# 启动第 1 个 vLLM 实例（端口 8001）在第一个终端上运行
(base) root@3e8799bf3e92:/data# CUDA_VISIBLE_DEVICES=0  vllm serve /mnt/moark-models/Qwen3-8B/ --port 8081 --host 0.0.0.0

# 启动第 2 个 vLLM 实例（端口 8002）在第二个终端上运行
(base) root@3e8799bf3e92:/data# CUDA_VISIBLE_DEVICES=1 vllm serve /mnt/moark-models/Qwen3-8B/ --port 8082 --host 0.0.0.0

# 验证资源隔离：查看每个 GPU 的使用情况
nvidia-smi  # 可看到 GPU 0 运行 7B 模型，GPU 1 运行 13B 模型，互不占用

```

### 14.4.3 资源隔离注意事项（官方强调）

- GPU 资源：避免多个模型共享同一 GPU（除非 GPU 显存充足），否则易导致 OOM 或性能急剧下降；

- CPU/内存资源：根据模型大小合理分配，7B 模型推荐至少 8 CPU 核心、16GB 内存，13B 模型推荐至少 12 CPU 核心、24GB 内存；

- 端口隔离：每个模型容器或实例需使用不同的端口，避免端口冲突。
