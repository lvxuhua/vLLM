## 10.1 PagedAttention 进阶：块大小选择、内存利用率监控

PagedAttention 是 vLLM 核心内存优化技术，通过将 KV 缓存分割为固定大小的块（Block），实现内存的高效分配与复用，解决传统 Attention 机制中内存碎片化问题。vLLM v0.7.0 对 PagedAttention 进行优化，支持自定义块大小、实时监控内存利用率，进一步提升内存使用效率。

### 10.1.1 块大小选择

块大小（Block Size）是 PagedAttention 的核心参数，指每个 KV 缓存块包含的 token 数量，直接影响内存利用率和推理性能，需根据模型大小、推理场景灵活选择，官方给出明确配置建议：

- 核心参数：`--block-size`（默认值：16，单位：token），取值范围为 8~128，需为 2 的幂（如 8、16、32、64、128）。

- 配置原则（官方推荐）：

    1. 小模型（≤7B）：推荐块大小 16~32，兼顾内存利用率和推理速度，避免块过大导致内存浪费；

    2. 大模型（≥13B）：推荐块大小 32~64，适配更长序列的 KV 缓存存储，减少块数量，降低内存分配开销；

    3. 超长序列推理（sequence_length ≥ 4096）：推荐块大小 64~128，进一步降低内存碎片化，提升缓存复用率。

实操示例（Bash 命令，配置块大小）：

```bash
# 1. 配置块大小
## 2. 启动API服务，配置块大小为32（适配13B模型）
(base) root@ac14c8d0800b:/data# vllm serve /mnt/moark-models/Qwen3-8B \
  --trust-remote-code \
  --dtype float16 \
  --block-size 32 \
  --max-model-len 2048 \
  --gpu-memory-utilization 0.9
  
```

## 10.2 自动前缀缓存（Automatic Prefix Caching）

自动前缀缓存是 vLLM 针对多请求场景的内存优化技术，核心原理是复用相同 prompt 前缀的 KV 缓存，避免重复计算和存储，大幅降低多并发场景下的内存占用，提升推理效率。vLLM  支持 V1 引擎专属的 zero-overhead 前缀缓存，进一步优化性能。

### 10.2.1 核心原理：相同前缀的 KV 缓存复用

当多个推理请求包含相同的 prompt 前缀（如相同的系统提示词、相同的问题开头）时，vLLM 会自动识别相同前缀，仅计算一次前缀对应的 KV 缓存，并将其复用给所有包含该前缀的请求，无需为每个请求单独计算和存储前缀 KV 缓存，从而减少内存占用和计算开销。

核心逻辑拆解：
      请求1输入 prompt：“请解释 vLLM 的 PagedAttention 技术”（前缀为“请解释 vLLM 的”）；vLLM 计算该 prompt 的 KV 缓存，存储前缀“请解释 vLLM 的”对应的 KV 缓存块；

请求2输入 prompt：“请解释 vLLM 的自动前缀缓存技术”（与请求1共享相同前缀）；vLLM 识别相同前缀，复用已存储的前缀 KV 缓存，仅计算新增部分（“自动前缀缓存技术”）的 KV 缓存；重复上述逻辑，所有共享相同前缀的请求均复用前缀 KV 缓存，降低内存占用。

### 10.2.2 适用场景与命中率优化

#### 10.2.2.1 适用场景

- 多并发推理场景：如 API 服务部署，多个用户发送包含相同系统提示词、相同前缀的请求；

- 固定前缀场景：如客服机器人（固定开场白前缀）、代码生成（固定导入语句前缀）、问答系统（固定指令前缀）；

- 长序列推理场景：相同前缀较长时，复用效果更明显，内存节省比例更高（最高可达 50% 以上）。

#### 10.2.2.2 命中率优化（官方实操建议）

前缀缓存命中率（复用的前缀缓存占总缓存的比例）直接影响内存优化效果，官方推荐以下优化方法：

1. 统一 prompt 前缀：尽量让多请求使用相同的系统提示词、指令前缀，减少前缀差异；
2. 调整前缀长度：前缀长度建议控制在 32~256 token，过长可能导致缓存复用率下降，过短则内存节省效果不明显；

### 10.2.3 V1 引擎中的 zero-overhead 前缀缓存

vLLM  V1 引擎（默认引擎）支持 zero-overhead 前缀缓存，核心优势是“复用缓存时无额外性能开销”，相比旧版引擎，避免了缓存复用过程中的数据拷贝、格式转换等开销，实现“内存节省+性能无损”。

官方补充：V1 引擎默认启用 zero-overhead 前缀缓存，无需额外配置，仅需确保使用 V1 引擎；若使用旧版引擎，需通过 `--engine v1` 参数切换。

```bash
# 确认启用V1引擎，开启zero-overhead前缀缓存
(base) root@ac14c8d0800b:/data# VLLM_LOGGING_LEVEL=DEBUG vllm serve /mnt/moark-models/Qwen3-8B \
> --dtype float16 \
> --block-size 32 \
> --max-model-len 2048 \
> --gpu-memory-utilization 0.9 \
> --enable_prefix_caching
```

## 10.3 KV 缓存量化：FP8、INT4 缓存压缩

KV 缓存是大模型推理过程中内存占用的主要部分（占总显存的 30%~60%），vLLM  支持对 KV 缓存进行量化压缩，通过 FP8、INT4 两种量化方式，在几乎不损失推理精度的前提下，大幅降低 KV 缓存的内存占用，适配更大模型、更长序列的推理场景。

### 10.3.1 核心原理与量化方式

KV 缓存量化的核心是通过降低 KV 缓存的数值精度，减少每个 token 对应的内存占用（如 FP16 为 2 字节/元素，FP8 为 1 字节/元素，INT4 为 0.5 字节/元素），同时通过量化校准，确保推理精度损失控制在可接受范围内（官方测试：精度损失≤3%）。

两种核心量化方式（官方对比）：

|量化方式|核心特点|内存节省比例|适用场景|
|---|---|---|---|
|FP8 量化|精度接近 FP16，推理精度损失极小（≤1%），无需复杂校准，兼容性强|50%（FP16→FP8）|对精度要求较高的场景（如问答、文本生成），兼顾内存与精度|
|INT4 量化|内存节省最多，精度损失略高（≤3%），需进行量化校准，适配部分模型|75%（FP16→INT4）|内存紧张、对精度要求不高的场景（如文本续写、批量生成）|
### 10.3.2 实操示例（Bash 命令，配置 KV 缓存量化）

```bash
# 1. 配置FP8 KV缓存量化（推荐，兼顾精度与内存）
(base) root@ac14c8d0800b:/data# VLLM_LOGGING_LEVEL=DEBUG vllm serve /mnt/moark-models/Qwen3-8B  --
dtype float16  --block-size 32  --max-model-len 2048  --gpu-memory-utilization 0.9  --enable_prefix_caching --kv-cache-dtype fp8

日志中显示「KV cache dtype: fp8」
```

## 10.4 内存利用率调优：gpu_memory_utilization、max_num_seqs

vLLM 提供两个核心参数，用于手动调优 GPU 内存利用率，分别是 `gpu_memory_utilization`（GPU 内存利用率阈值）和 `max_num_seqs`（最大并发序列数），通过合理配置这两个参数，可实现内存利用率与推理性能的平衡，避免显存溢出或内存浪费。

### 10.4.1 核心参数解析（官方标准定义）

1. **gpu_memory_utilization（GPU 内存利用率阈值）**参数作用：指定 vLLM 可使用的 GPU 内存比例，取值范围为 0.1~1.0（默认值：0.9，即 90%），控制模型权重、KV 缓存等占用的 GPU 内存上限。配置原则（官方推荐）：

    - 内存充足（GPU 显存 ≥ 24GB，部署 7B 模型）：推荐 0.9~0.95，最大化利用内存；

    - 内存紧张（GPU 显存 16GB，部署 7B 模型）：推荐 0.7~0.8，预留部分内存用于 KV 缓存扩展；

    - 多模型部署：推荐 0.6~0.7，为其他模型预留足够内存。

2. **max_num_seqs（最大并发序列数）**参数作用：指定 vLLM 同时处理的最大推理序列数（每个序列对应一个推理请求），直接影响内存占用和并发性能。配置原则（官方推荐）：
          

    - 小模型（≤7B）+ 大显存（≥24GB）：推荐 max_num_seqs=32~64；

    - 大模型（≥13B）+ 中显存（16~24GB）：推荐 max_num_seqs=8~16；

    - 内存紧张或超长序列：推荐 max_num_seqs=4~8，避免显存溢出。

### 10.4.2 实操示例（Bash 命令，调优内存利用率）

```bash
# 1. 内存充足场景（24GB GPU，7B模型），最大化内存利用率
(base) root@ac14c8d0800b:/data# VLLM_LOGGING_LEVEL=DEBUG vllm serve /mnt/moark-models/Qwen3-8B  --dtype float16  --block-size 32  --max-model-len 2048  --gpu-memory-utilization 0.5  --enable_prefix_caching  --max-num-seqs 10

```
