# 12.1 量化概述

在大模型推理部署中，量化是平衡模型性能、显存占用与推理效率的核心优化技术。vLLM 作为高性能大模型推理引擎，全面支持多种量化格式与量化方法，可针对不同硬件环境、业务场景（高吞吐量、低延迟、显存受限等）提供灵活的量化解决方案，帮助开发者在有限硬件资源下实现大模型的高效部署。

vLLM 的量化设计遵循“精度无损优先、性能最优适配”的原则，既支持权重量化、KV 缓存量化，也支持多模态模型量化，兼容主流量化算法（GPTQ、AWQ 等）与硬件平台（NVIDIA GPU、Ascend 等），同时提供完善的精度与性能评估工具，降低量化部署的门槛。

## 12.1.1 量化的基本概念

量化本质是将模型训练与推理过程中使用的高精度数据（如 FP32、FP16），映射为低精度数据（如 FP8、INT8、INT4）的过程，核心是通过牺牲极小的精度损失，换取显存占用的大幅降低与推理速度的提升。

从量化对象来看，vLLM 支持两类核心量化场景：

1. **权重量化**：对模型的权重参数进行量化，是最常用的量化方式，可直接降低模型本身的显存占用（通常占模型总显存开销的 60%-80%）。

2. **激活值与 KV 缓存量化**：对模型推理过程中生成的中间激活值、KV 缓存进行量化，进一步降低推理过程中的峰值显存占用，尤其适用于超长上下文推理场景。

从量化时机来看，vLLM 支持两种量化模式：

1. **离线量化**：在模型加载前，通过量化工具（如 GPTQ-for-LLaMa、AWQ 工具）对模型权重进行量化，生成量化后的模型文件，vLLM 直接加载量化模型进行推理，适合生产环境部署。

2. **在线量化**：在模型加载过程中，通过 vLLM 内置的量化接口（如 BitsAndBytes 集成）动态对模型进行量化，无需提前处理模型文件，适合快速测试与验证场景。

## 12.1.2 量化的收益与代价

### 1. 核心收益

1. **降低显存占用**：这是量化最核心的收益。例如，INT8 量化可将 FP16 模型的显存占用降低 50%，INT4 量化可降低 75%，FP8 量化可降低 50%，使大模型（如 70B、175B 参数模型）能够在单张或少量 GPU 上部署。

2. **提升推理吞吐量**：低精度数据的计算（如 INT8、FP8）可充分利用 GPU 的张量核心（Tensor Core），减少内存带宽消耗，提升单位时间内的推理请求处理数量，尤其适合批量推理场景。

3. **降低部署成本**：显存占用的降低意味着无需使用高端大容量 GPU（如 A100、H100），可使用中端 GPU（如 A10、RTX 3090）部署大模型，大幅降低硬件采购成本。

### 2. 潜在代价

1. **精度损失**：量化过程中，高精度数据映射为低精度数据会导致部分信息丢失，可能造成模型推理精度下降（如困惑度升高、生成文本质量降低、多模态任务准确率下降）。不同量化格式的精度损失差异较大，FP8 精度损失极小（≤3%），INT8 精度损失轻微（≤5%），INT4 精度损失中等（5%-10%）。

2. **量化校准开销**：为降低精度损失，量化过程中需要通过校准数据进行参数校准（如缩放因子校准），会增加一定的预处理时间，尤其离线量化需要额外的量化工具链操作。

3. **硬件兼容性限制**：部分低精度量化格式（如 FP8）需要特定硬件支持（如 NVIDIA Hopper 架构、Ampere 架构），老旧 GPU 可能无法支持，限制了量化方案的选择。

vLLM 量化的核心目标的是“最小化精度损失，最大化性能收益”，通过优化量化算法、校准策略与硬件适配，平衡收益与代价，满足不同业务场景的需求。

# 12.2 精度类型介绍

vLLM 全面支持多种量化精度格式，涵盖浮点型（FP8）与整型（INT8、INT4 等），不同精度格式在显存占用、精度损失、硬件兼容性上存在显著差异，为后续量化方法的选择与配置奠定基础。本节将系统梳理 vLLM 支持的核心精度类型，明确各类精度的技术细节与适用场景。

## 12.2.1 FP8 精度（E4M3 / E5M2）

FP8 是 NVIDIA 推出的 8 位浮点精度格式，分为两种子类型（E4M3、E5M2），是兼顾精度与性能的高端量化格式，vLLM 对其提供完整支持，适用于对精度要求较高、需要节省显存的场景。

### 1. 两种子类型技术细节

1. **E4M3（4 位指数 + 3 位尾数）**：
        

    - 指数位 4 位，可表示的指数范围为 -8~+7；

    - 尾数位 3 位，精度高于 E5M2，适合对精度敏感的场景；

    - 支持的数值范围：~1.4e-45 ~ ~1.7e38，可覆盖 FP16 的核心数值范围，几乎无溢出风险。

2. **E5M2（5 位指数 + 2 位尾数）**：
        

    - 指数位 5 位，可表示的指数范围为 -16~+15；

    - 尾数位 2 位，精度略低于 E4M3，但数值范围更广；

    - 适合中间激活值、KV 缓存等对数值范围要求较高的场景。

### 2. vLLM 中的 FP8 支持特性

1. 同时支持 E4M3 与 E5M2 两种子类型，可通过配置参数指定；

2. 支持权重量化、KV 缓存量化与激活值量化，可单独或联合使用；

3. 精度损失极轻微（相对 FP16，精度损失 ≤3%），几乎不影响模型推理质量；

4. 可充分利用 NVIDIA GPU 的张量核心，推理性能接近 FP16，同时显存占用降低 50%。

### 3. 适用场景

高端 GPU 部署、对精度要求高的场景（如对话生成、多模态推理）、需要平衡精度与显存的大模型（如 70B 参数模型）。

## 12.2.2 INT8 精度

INT8 是最常用的 8 位整型量化格式，也是 vLLM 支持最完善的量化格式之一，核心是将 FP16/FP32 数据映射到 [-128, 127] 的整型范围，通过缩放因子校准降低精度损失，是兼顾精度、性能与兼容性的通用量化方案。

### 1. 核心技术细节

1. 采用对称量化或非对称量化策略，vLLM 默认采用对称量化（适合权重参数），非对称量化可通过配置启用（适合激活值）；

2. 通过校准数据计算缩放因子（scale），将高精度数据归一化到 INT8 范围，确保量化误差最小；

3. 支持 per-tensor 量化（单一张量使用一个缩放因子）与 per-channel 量化（每个通道使用一个缩放因子），vLLM 优先采用 per-channel 量化，精度更高。

### 2. vLLM 中的 INT8 支持特性

1. 全面支持权重量化与 KV 缓存量化，激活值量化需结合特定量化方法（如 GPTQ）；

2. 精度损失轻微（相对 FP16，精度损失 ≤5%），适用于大多数文本生成、问答等场景；

3. 硬件兼容性强，支持 NVIDIA Ampere 及以上架构 GPU（如 A10、A100、RTX 30 系列、RTX 40 系列），无需高端硬件；

4. 推理性能提升显著，相比 FP16，吞吐量提升 1.5~2 倍，显存占用降低 50%。

### 3. 适用场景

通用大模型部署、中端 GPU 环境、对精度要求中等的场景（如批量文本生成、智能客服），是 vLLM 量化部署的首选方案。

## 12.2.3 INT4 精度

INT4 是 4 位整型量化格式，是显存节省效果最显著的量化格式，核心是将 FP16/FP32 数据映射到 [-8, 7] 的整型范围，适合显存极度受限、对精度要求不高的场景。

### 1. 核心技术细节

1. 采用 per-group 量化策略（将权重张量分为多个组，每组使用一个缩放因子），平衡精度与显存占用；

2. 量化过程中需要更精细的校准（如使用更多校准数据），否则易出现明显精度损失；

3. 支持稀疏量化优化，进一步降低显存占用，同时减少计算开销。

### 2. vLLM 中的 INT4 支持特性

1. 主要支持权重量化，KV 缓存 INT4 量化需结合特定优化（如 Marlin 内核）；

2. 精度损失中等（相对 FP16，精度损失 5%-10%），部分场景（如摘要生成、简单问答）可满足需求；

3. 显存节省效果显著，相比 FP16，显存占用降低 75%，可在单张中端 GPU（如 A10 24GB）部署 70B 参数模型；

4. 需依赖优化内核（如 Marlin）提升推理性能，否则可能出现性能下降。

### 3. 适用场景

显存受限的部署环境（如单张 GPU 部署大模型）、对精度要求不高的场景（如文本摘要、关键词提取）、批量低精度推理场景。

## 12.2.4 其他精度格式

除上述核心精度格式外，vLLM 还支持多种其他量化精度格式，适配不同的量化算法与业务场景，主要包括以下几种：

1. **NF4（Normalized Float4）**：
        

    - 4 位浮点量化格式，专为大模型权重量化设计，通过归一化处理降低精度损失；

    - 精度优于 INT4，接近 INT8，显存占用与 INT4 一致（降低 75%）；

    - vLLM 主要通过 BitsAndBytes 集成支持，适合对精度要求高于 INT4 的场景。

2. **FP4**：
        

    - 4 位浮点量化格式，与 INT4 显存占用相当，但精度略高；

    - 适合对数值范围敏感的场景（如激活值量化），vLLM 仅在特定量化方法（如 AWQ）中支持。

3. **BF16**：
        

    - 16 位浮点量化格式（本质是 FP32 的简化版），显存占用是 FP32 的 50%，精度接近 FP32；

    - 不属于严格意义上的低精度量化，但 vLLM 将其作为过渡方案，适合对精度要求极高、不接受 INT8/FP8 精度损失的场景；

    - 硬件兼容性强，支持 NVIDIA Ampere 及以上架构 GPU。

## 12.2.5 硬件兼容性要求

不同精度格式对 GPU 硬件架构存在明确要求，vLLM 仅在兼容的硬件上支持对应精度的量化推理，以下是核心精度格式的硬件兼容性汇总：

# 12.3 权重量化

权重量化是 vLLM 量化推理的核心场景，针对模型权重参数进行低精度转换，直接降低模型的基础显存占用。vLLM 支持多种主流权重量化方法，包括 GPTQ、AWQ、BitsAndBytes 等，同时集成 Marlin 内核加速低精度推理，适配不同精度格式（INT4、INT8、FP8 等）。

## 12.3.1 GPTQ 量化

GPTQ 是一种高效的权重量化算法，核心是通过梯度下降优化量化误差，在 INT4/INT8 精度下实现接近 FP16 的推理精度，是 vLLM 推荐的离线量化方案之一，支持大多数主流大模型（如 LLaMA、ChatGLM、Qwen 等）。

### 1. 核心原理

GPTQ 采用“逐层量化 + 梯度优化”策略，针对每个 Transformer 层的权重张量，通过以下步骤完成量化：

1. 将权重张量分为多个小组（如 128 个元素为一组）；

2. 对每组权重进行量化（INT4/INT8），计算量化误差；

3. 通过梯度下降调整量化参数（如缩放因子），最小化量化误差；

4. 重复上述步骤，直至所有层量化完成，生成量化权重文件。

### 2. vLLM 中的 GPTQ 支持特性

1. 支持 INT4、INT8 两种精度的 GPTQ 量化。

2. 支持 per-channel 量化与 per-group 量化，默认采用 per-group 量化（组大小 128），平衡精度与性能；

3. 支持量化模型的快速加载，无需重新校准，直接部署推理；

4. 兼容 Hugging Face Hub 上的 GPTQ 量化模型，可直接通过模型名称加载（如 `TheBloke/Llama-2-7B-GPTQ`）。

## 12.3.2 AWQ 量化

AWQ（Activation-aware Weight Quantization）是另一种高效的权重量化算法，核心是结合激活值分布优化权重量化，相比 GPTQ，在 INT4 精度下具有更高的推理精度，同时推理速度更快，vLLM 对其提供完整支持。

### 1. 核心原理

AWQ 与 GPTQ 的核心区别的是，量化过程中不仅考虑权重本身的分布，还结合激活值的分布进行优化，具体步骤如下：

1. 通过校准数据获取模型各层的激活值分布；

2. 根据激活值分布，确定权重的重要性（对激活值影响较大的权重保留更高精度）；

3. 对权重进行分组量化，对重要权重采用更精细的量化策略，降低量化误差；

4. 生成量化权重文件，支持快速加载与推理。

### 2. vLLM 中的 AWQ 支持特性

1. 支持 INT4、FP4 两种精度的 AWQ 量化，INT4 精度为主要应用场景；

2. 精度优于 GPTQ，相同 INT4 精度下，困惑度更低，生成文本质量更高；

3. 推理速度比 GPTQ 快 10%-20%，可充分利用 GPU 张量核心；

4. 兼容 Hugging Face Hub 上的 AWQ 量化模型，支持一键加载。

## 12.3.3 BitsAndBytes 集成

BitsAndBytes 是一种轻量级在线量化工具，支持在模型加载过程中动态对权重进行量化，无需提前生成量化模型文件，适合快速测试与验证量化效果，vLLM 对其提供无缝集成支持。

### 1. 核心特性

1. **在线量化**：无需离线量化步骤，加载模型时直接对权重进行量化，快速验证量化效果；

2. **支持多精度**：支持 INT4、INT8、NF4 三种精度，其中 NF4 精度是 BitsAndBytes 的特色，精度优于 INT4；

3. **轻量级集成**：仅需安装 BitsAndBytes 库，通过 vLLM 简单参数配置即可启用；

4. **兼容性广**：支持所有 Hugging Face 格式的大模型，无需对模型进行修改。

### 2. vLLM 中的 BitsAndBytes 配置方式

```
安装依赖：
pip install bitsandbytes

启动 vLLM 服务（在线量化）：
    `# INT4

# INT8 量化
(base) root@9c74b76a6bea:~# vllm serve /mnt/moark-models/Qwen3-0.6B/  --quantization fp8

```

### 3. 优缺点与适用场景

- **优点**：无需离线量化，快速验证；操作简单，无需专业量化工具；兼容所有大模型。

- **适用场景**：量化效果快速验证、科研测试、临时部署场景，不推荐用于生产环境。

# 12.4 KV 缓存量化

KV 缓存是大模型推理过程中显存占用的重要组成部分，尤其在超长上下文推理场景中，KV 缓存的显存占用甚至超过模型权重本身。vLLM 支持对 KV 缓存进行量化，进一步降低推理过程中的峰值显存占用，同时通过优化校准策略，最小化精度损失。

vLLM 目前主要支持 FP8 KV 缓存量化，同时支持 INT8 KV 缓存量化，核心是通过缩放因子校准，将 FP16 格式的 KV 缓存转换为低精度格式，不影响推理流程与生成质量。

## 12.4.1 FP8 KV 缓存

FP8 KV 缓存是 vLLM 推荐的 KV 量化方案，兼顾精度与性能，相比 FP16 KV 缓存，显存占用降低 50%，精度损失极轻微（≤3%），适合大多数推理场景，尤其是超长上下文场景。

### 1. 核心技术细节

1. 采用 E5M2 子类型 FP8 格式（数值范围更广，适合 KV 缓存的激活值分布）；

2. 通过在线校准方式，实时计算 KV 缓存的缩放因子，确保量化误差最小；

3. 支持 per-sequence 校准（每个推理序列使用独立的缩放因子），适配不同序列的激活值分布；

4. 与权重量化协同工作，可同时启用 FP8 权重量化与 FP8 KV 缓存量化，最大化显存节省效果。

### 2. vLLM 中的 FP8 KV 缓存启用方式

FP8 KV 缓存需结合权重量化启用，支持命令行与 Python 代码两种配置方式，严格参照官方文档：

命令行配置（FP8 权重量化 + FP8 KV 缓存）：

```
(base) root@ac14c8d0800b:/data# VLLM_LOGGING_LEVEL=DEBUG vllm serve /mnt/moark-models/Qwen3-8B  --
dtype float16  --block-size 32  --max-model-len 2048  --gpu-memory-utilization 0.9  --enable_prefix_caching --kv-cache-dtype fp8
```



### 3. 适用场景

超长上下文推理场景（如 8k、16k 上下文）、高并发推理场景、对精度要求较高的 KV 缓存量化场景，是 vLLM KV 缓存量化的首选方案

## 12.3.3 配置参数详解

vLLM 提供多个 KV 缓存量化相关参数，支持精细化配置，以下是核心参数（严格参照 vLLM 官方文档）：

1. `--kv_cache_dtype`：
        

    - 作用：指定 KV 缓存的量化精度格式；

    - 可选值：`fp16`（默认，不量化）、`fp8`、`int8`；

    - 注意：`int8` 仅支持部分模型，需结合 GPTQ 量化使用。
4. `--kv_offloading_backend`：
        

    - 作用：结合 KV 缓存量化与 CPU 卸载，进一步降低 GPU 显存占用；

    - 可选值：`native`、`cpu`；

    - 使用场景：显存极度受限的场景，需与 `--kv_cache_dtype` 配合使用。
